## 关键经验

### 1. PostgreSQL密码陷阱
**问题**: deploy-xiaoqing.py 中硬编码了错误的DB密码 `zuoyelang2024`，实际容器密码是 `uoUSiW6cmCxrtKTY21bp`。
**诊断**: Prisma报 `P1001: Can't reach database server`，但端口测试显示OPEN。真正原因是认证失败，不是网络不通。
**铁律**: Prisma的P1001错误不一定是网络问题，可能是认证失败。用 `docker inspect` 查看容器 POSTGRES_PASSWORD 环境变量确认真实密码。

### 2. Docker Nginx端口暴露
**问题**: 部署脚本写了 conf.d/zuting.conf 监听3080/3083端口，但Docker容器只暴露80/443。
**结果**: 端口永远返回000 (connection refused)。
**铁律**: Docker容器内nginx监听的端口必须在 `-p` 映射中声明才能从宿主机访问。域名路由应写在主nginx.conf的port 80 server block中。

### 3. Docker内proxy_pass地址
**问题**: conf.d文件中用 `proxy_pass http://127.0.0.1:3003` — 在容器内 127.0.0.1 指向容器自身，不是宿主机。
**解决**: 使用Docker bridge gateway IP `172.19.0.1` 访问宿主机上的node进程。
**获取方式**: `docker network inspect bridge --format '{{(index .IPAM.Config 0).Gateway}}'`

### 4. sed在nginx配置中的危险性
**问题**: 用sed修改nginx-standalone.conf时，正则匹配过于宽泛，导致在多处插入了损坏的配置行(`nclude`而非`include`)。
**铁律**: 修改nginx配置时，用sftp直接写入完整文件，不要用sed做局部修改。关键配置文件只能整体替换。

### 5. Admin代码分割策略
- manualChunks分离: react(36KB) / antd(965KB) / recharts(407KB)
- React.lazy()按路由懒加载: 12个页面各1-6KB
- 初始加载从502KB gzip降至77KB gzip
- antd仍是最大chunk但作为并行加载不阻塞首屏

### 6. npm install部署策略
**问题**: 之前用 `npm install --omit=dev @nestjs/core@11 prisma@7 ...` 显式指定包+版本，导致eresolve冲突。
**解决**: 直接 `npm install --legacy-peer-deps`，让npm读package.json自动解析。不要用 `--omit=dev` 因为prisma/tsx/swc在devDeps中但deploy需要。
