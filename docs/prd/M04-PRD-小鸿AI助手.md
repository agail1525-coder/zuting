# M04 小鸿AI助手 产品需求文档

> 版本: v1.0 | 日期: 2026-03-25 | 状态: 已发布

---

## 1. 需求背景与目标

### 1.1 业务背景

小鸿（Xiaohong，"Little Hong"）是祖庭旅行平台的AI灵性向导。作为平台的智能交互核心，小鸿具备跨信仰知识体系，能够回答用户关于 12 大信仰、60 处圣地、27 座祖庭、28 位祖师、39 条祖训、30 枚印的各类问题，并提供修行建议、旅行推荐和文化解读。

小鸿的定位不是通用 AI 聊天机器人，而是一位深谙全球宗教文化的灵性伙伴。其回答风格温和、智慧、包容，尊重每一种信仰传统，同时能以现代语言为用户提供清晰的指引。

**技术实现：**
- 后端通过 NestJS xiaohong 模块提供 API
- 支持两种模式：同步回复（POST /chat）和流式回复（GET /chat/stream, SSE）
- Web 端已接入真实 API，App 端和小程序端可能仍使用 mock 数据
- 快速提问建议通过 GET /suggestions 获取

### 1.2 需求目标（量化）

| 指标 | 目标值 | 度量方式 |
|------|--------|----------|
| 聊天功能可用率 | > 99% | API 健康检查 |
| 首条回复延迟 | SSE 首 token < 1s, 同步回复 < 3s | API 监控 |
| 用户满意度 | 对话后正面反馈率 > 70% (v2) | 用户评价 |
| 日均对话数 | > 200 次 | 聊天记录统计 |
| 建议点击率 | suggestions 点击率 > 25% | 埋点分析 |
| 流式体验流畅度 | SSE 中断率 < 1% | 前端错误监控 |

### 1.3 需求范围

**v1 包含：**
- AI 聊天界面（消息气泡、输入框、发送按钮）
- SSE 流式回复（打字机效果）
- 同步回复（POST 模式，降级方案）
- 快速提问建议
- Web / App / 小程序三端聊天页
- 管理后台 AI 配置页

**v2 规划（本期不含）：**
- 语音输入
- 对话历史保存与回溯
- 上下文感知推荐（基于用户浏览历史）
- 多轮对话记忆
- 聊天导出/分享
- 用户对 AI 回答的评价（赞/踩）

## 2. 用户故事

### US-01 向小鸿提问

**作为** 一名对佛教禅修感兴趣的用户，
**我希望** 在聊天页面输入问题后，小鸿能以流式方式逐字回复，
**以便** 我能获得即时的、自然的对话体验。

**验收标准：**
- [ ] 输入框位于页面底部，支持回车发送
- [ ] 用户消息显示为右侧气泡（深色背景）
- [ ] AI 回复显示为左侧气泡（深灰背景），带小鸿金色头像
- [ ] SSE 流式回复：文字逐字/逐词出现（打字机效果）
- [ ] 回复中显示打字光标动画
- [ ] 回复完成后光标消失

### US-02 使用快速提问建议

**作为** 一名初次使用的用户，不知道该问什么，
**我希望** 看到预设的快速提问建议按钮，点击即可发送，
**以便** 我能快速开始与小鸿的对话。

**验收标准：**
- [ ] 聊天页打开时，如果没有历史消息，显示欢迎语 + 建议按钮
- [ ] 建议按钮 3-5 个，内容来自 GET /suggestions API
- [ ] 点击建议按钮等同于输入该文字并发送
- [ ] 发送后建议按钮区域隐藏或收起
- [ ] 建议内容涵盖不同信仰和主题

### US-03 流畅的多轮对话

**作为** 一名正在深入探讨某话题的用户，
**我希望** 连续提问时，聊天窗口自动滚动到最新消息，
**以便** 我能流畅地进行多轮对话。

**验收标准：**
- [ ] 新消息出现时自动滚动到底部
- [ ] SSE 流式输出过程中持续滚动
- [ ] 上滑可查看历史消息（本次会话内）
- [ ] 用户主动上滑时暂停自动滚动
- [ ] 出现新消息后底部显示"回到最新"浮动按钮

### US-04 处理网络异常

**作为** 一名网络状况不佳的用户，
**我希望** 在网络中断时看到清晰的错误提示和重试选项，
**以便** 我不会对聊天功能失去信心。

**验收标准：**
- [ ] SSE 连接中断时显示"网络中断，正在重连..."
- [ ] 自动重试 3 次，间隔递增（1s, 2s, 4s）
- [ ] 重试失败后降级为同步 POST 模式
- [ ] POST 也失败时显示"发送失败" + 红色标记 + [重试] 按钮
- [ ] 已发送但无回复的消息显示"等待回复..."状态

### US-05 管理员配置 AI

**作为** 管理员，
**我希望** 在后台配置小鸿的人格设定、知识库和敏感词，
**以便** 我能控制 AI 的回答质量和安全性。

**验收标准：**
- [ ] AI 配置页包含：人格描述（TextArea）、系统提示词（TextArea）
- [ ] 知识库管理：可添加/编辑/删除知识条目
- [ ] 敏感词/屏蔽词列表管理
- [ ] 快速建议管理：编辑 suggestions 列表
- [ ] 配置保存后实时生效

### 3.2 业务规则清单

| 规则编号 | 描述 | 触发条件 | 处理逻辑 | 异常处理 |
|----------|------|----------|----------|----------|
| R-01 | 消息长度限制 | 用户输入消息 | 最大 2000 字符 | 超出时输入框禁止输入 + 字数提示 |
| R-02 | 空消息禁止发送 | 点击发送 | trim 后为空则不发送 | 发送按钮置灰 |
| R-03 | 发送频率限制 | 连续发送 | 相邻两次发送间隔 ≥ 1s | 按钮短暂禁用 |
| R-04 | SSE 超时 | 流式回复 | 30s 无新 token 则判定超时 | 关闭连接，显示"回复超时" |
| R-05 | SSE 优先策略 | 发送消息 | 优先使用 SSE，失败后降级 POST | 见降级流程 |
| R-06 | 敏感词过滤 | AI 回复/用户输入 | 后端检查敏感词列表，命中则替换为 *** | 后端处理 |
| R-07 | 建议数量 | GET /suggestions | 返回 3-5 条建议 | 接口失败则隐藏建议区域 |
| R-08 | 回复中禁止再次发送 | AI 正在回复 | 发送按钮禁用，显示 loading | 回复完成后恢复 |
| R-09 | 会话内消息上限 | 单次会话 | 最多 100 条消息（50 轮对话） | 达上限提示"对话已满，请开始新对话" |
| R-10 | 小鸿人格一致性 | 所有回复 | 回复风格温和、智慧、包容 | 通过 system prompt 控制 |

## 5. 数据模型

### 5.1 实体关系

v1 阶段小鸿模块为无状态设计（不持久化聊天记录），消息仅存在于前端会话内存中。

v2 引入持久化后的实体关系：

```
User (1) ──────< (N) Conversation (v2)
Conversation (1) ──────< (N) Message (v2)
AIConfig (singleton) ──── 全局配置
```

### 5.2 核心字段

**Message（前端模型，v1 仅内存）：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 前端生成的唯一 ID (uuid) |
| role | 'user' \| 'assistant' | 是 | 消息角色 |
| content | string | 是 | 消息内容 |
| timestamp | Date | 是 | 发送/接收时间 |
| status | 'sending' \| 'sent' \| 'streaming' \| 'done' \| 'error' | 是 | 消息状态 |
| errorMessage | string | 否 | 错误信息（status=error 时） |

**AIConfig（后端模型，管理后台配置）：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | String | 是 | 主键（单例） |
| persona | String | 是 | 人格描述，如"温和智慧的灵性向导" |
| systemPrompt | String | 是 | 系统提示词（发送给 LLM 的 system message） |
| knowledgeBase | JSON | 否 | 知识库条目列表 |
| bannedWords | String[] | 否 | 敏感词/屏蔽词列表 |
| suggestions | String[] | 否 | 快速建议列表 |
| maxTokens | Int | 否 | 单次回复最大 token 数，默认 1024 |
| temperature | Float | 否 | LLM temperature 参数，默认 0.7 |
| updatedAt | DateTime | 是 | 最后更新时间 |

**Conversation（v2 持久化模型）：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | String (cuid) | 是 | 主键 |
| userId | String (FK) | 是 | 用户 ID |
| title | String | 否 | 对话标题（取首条消息前 20 字） |
| createdAt | DateTime | 是 | 创建时间 |
| updatedAt | DateTime | 是 | 更新时间 |

**ChatMessage（v2 持久化模型）：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | String (cuid) | 是 | 主键 |
| conversationId | String (FK) | 是 | 对话 ID |
| role | String | 是 | 'user' 或 'assistant' |
| content | String | 是 | 消息内容 |
| rating | Int | 否 | 用户评价: 1=赞, -1=踩 (v2) |
| createdAt | DateTime | 是 | 创建时间 |

## 6. API 接口

| 接口 | Method | 路径 | 入参 | 出参 | 说明 |
|------|--------|------|------|------|------|
| 同步聊天 | POST | /api/xiaohong/chat | { message: string } | { reply: string } | 完整回复，降级方案 |
| 流式聊天 | GET | /api/xiaohong/chat/stream | ?message=string | SSE stream (data: {token}) | 逐 token 流式输出 |
| 快速建议 | GET | /api/xiaohong/suggestions | — | string[] | 3-5 条建议文本 |
| 获取 AI 配置 | GET | /api/xiaohong/config | — | AIConfig | 管理员权限 |
| 更新 AI 配置 | PATCH | /api/xiaohong/config | AIConfigUpdateDto | AIConfig | 管理员权限 |

**POST /xiaohong/chat 请求体：**
```json
{
  "message": "string (required, max 2000 chars)"
}
```

**POST /xiaohong/chat 响应体：**
```json
{
  "reply": "你好！我是小鸿，祖庭旅行平台的灵性向导。很高兴为你服务..."
}
```

**GET /xiaohong/chat/stream SSE 事件格式：**
```
data: {"token":"你好"}

data: {"token":"！"}

data: {"token":"我是"}

data: {"token":"小鸿"}

...

data: {"token":"[DONE]"}
```

**GET /xiaohong/suggestions 响应体：**
```json
[
  "佛教的核心教义是什么？",
  "推荐几个适合初次朝圣的圣地",
  "曹溪三十印的修行次第是怎样的？",
  "道教和佛教有什么不同？",
  "如何规划一次祖庭之旅？"
]
```

**AIConfigUpdateDto：**
```json
{
  "persona": "string",
  "systemPrompt": "string",
  "knowledgeBase": [{"topic": "string", "content": "string"}],
  "bannedWords": ["string"],
  "suggestions": ["string"],
  "maxTokens": "number (100-4096)",
  "temperature": "number (0-2)"
}
```

### 7.3 小程序端（Taro）

#### 页面 MP-01: 聊天页 `/chat`

**入口：** 底部 Tab / 首页入口

**布局：** 与 App 类似

**与 Web/App 差异：**
- SSE 不支持（微信小程序不支持 EventSource），固定使用 POST 同步模式
- 回复为完整显示（无打字机效果），可添加简单的文字逐字展示动画（setInterval 模拟）
- 输入框使用 Taro textarea 组件
- 建议按钮使用 grid 布局（2x2）
- 支持微信分享对话截图（v2）
- 键盘弹出处理使用 Taro 内置方案

**小程序特殊处理：**
- 由于 POST 同步模式，需要显示明确的 loading 态（旋转图标 + "小鸿正在思考..."）
- 设置合理的超时时间（15s），超时后提示用户
- 消息气泡内文字使用 rich-text 组件（支持简单 HTML 格式化）
