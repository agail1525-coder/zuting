## 1. 升级背景
v1 (P37.04 原版) 12 路答案均为字符串占位 `[ZEN 视角的答案占位 — Wave 2 接入小鸿 LLM]`。
v2 目标：**小鸿深度介入** —— 以 12 个宗师分身调用 Qwen3.5-35B，每位答复必须依据自身教义/经典，加入讨论模式让一问变多观点，用户快速建立信仰。

## 6. API 变更
### 6.1 POST /api/cultivation/wisdom/query （升级）
**Body**:
```json
{ "question": "string 5-500", "traditions": ["ZEN","TAOISM",...] }
```
- `traditions` 长度 3-12；未传则取 primary+blend 前 6
- 服务端并行（Promise.allSettled）：per master → tradition-filtered scripture RAG → LLM 第一人称作答 → 结构化返回
- **单 master 失败降级**：`status: "FAILED", answer: "此刻大师沉默不语"`，不拖整单

**Response**: `WisdomQuery`，`answers[]` 结构：
```ts
{
  tradition: "ZEN",
  masterName: "惠能",
  answer: "300-500 字第一人称",
  citations: [{ scriptureSlug, title, quote }],
  keyPoints: ["破执","见性"],
  status: "OK" | "FAILED"
}
```

### 6.2 POST /api/cultivation/wisdom/synthesize （升级）
- 读取 chosen answers → 一次 LLM 调用（"融通者小鸿"元角色）
- 返回结构化 synthesis（存 JSON.stringify 到 `synthesized` 字段，不改 schema）：
```ts
{
  convergence: string[],   // 共识 3-5
  divergence: Array<{ tradition, stance, reason }>,
  integration: string,     // 融通建议 500 字
  practice: string         // 可执行今日行动
}
```

### 6.3 POST /api/cultivation/wisdom/:id/debate （新）
- **Body**: `{ rounds?: 3, includeModerator?: true }`
- **Schema 变更**：`WisdomQuery.debate Json?`（`prisma db push`）
- **逻辑**：基于已有 `answers` 作为 round 0，逐轮让每位 master 读取其他 master 上一轮答复，呼应/补充/温和反驳，继续引用自宗经论
- **Moderator**：可选小鸿元角色每轮后做"共鸣/张力"一句话总结
- **限流**：`@Throttle(5/h)` （多轮更贵）

**Response**:
```ts
debate: {
  rounds: [
    { round: 1, turns: [{ tradition, masterName, response, citations, repliesTo: string[] }] },
    { round: 2, turns: [...] },
    { round: 3, turns: [...] }
  ],
  moderatorNotes?: string[]
}
```

## 7. Prompt 工程
### 7.1 答复 system prompt
```
你是「{masterName}」，{persona.personality}。
你的核心关切：{coreConcern}
你的写作风格：{writingStyle}
你的标志性词汇：{signatureVocab.join('、')}

回答原则：
1. 第一人称，像宗师亲自对话
2. 必须引用自宗经论至少 1 条，格式：《经论名·品名》"原文节选"（20-60 字）
3. 答复 300-500 字，不用 Markdown 标题
4. 不得出现"我是AI"等破戒用语
5. 不得引用其他宗派经典（保持本宗立场的纯粹）

---
检索到的本宗经论上下文（优先引用）：
{scriptureContext}

---
返回 JSON:
{
  "answer": "第一人称答复",
  "citations": [{"scriptureSlug":"","title":"","quote":""}],
  "keyPoints": ["要点1","要点2"]
}
```

### 7.2 讨论 system prompt 增量
在答复 prompt 基础上追加：
```
这是圆桌讨论第 {k} 轮。你刚才已经给出了你的观点。
其他大师上一轮的答复：
{otherMastersLastTurn}

现在请：
1. 呼应 2 位以上让你共鸣的大师的观点
2. 如有必要，**温和地**指出你认为需要补充或调整的地方（不得轻贬他宗）
3. 结合本宗经论进一步深化你的立场
4. 在 citations 外新增 repliesTo: [tradition...]，标记你此轮呼应的大师
```

### 7.3 融合 synthesizer prompt
```
你是"融通者小鸿"，不代表任何单一传统。用户提问：{question}
{N} 位大师的答复摘录：
{answersSummary}

请客观提炼：
- convergence: 3-5 条共识
- divergence: 每条分歧写 {tradition, stance, reason}
- integration: 500 字整合建议（不偏袒任何一宗）
- practice: 一条用户今日可执行的行动
```

## 12. 风险与对策
| 风险 | 对策 |
|---|---|
| 12 并发 LLM 超时 | 默认 6 位 + 用户扩展；per-master 30s 超时；失败降级 |
| 引用幻觉（编造 scripture slug） | service 层过滤未知 slug；prompt 强调"只能从候选清单中选" |
| 宗派偏见/不平等引用 | 每位 master 有独立 RAG；融合 prompt 强调中立 |
| 敏感问题 | 复用 moderation 模块前置拦截 |
| 成本 | Redis 缓存相同问题 7 天（v1 已规划，延用） |
