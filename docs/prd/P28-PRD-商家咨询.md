# P28 商家咨询 产品需求文档

> 版本: v1.0 | 日期: 2026-03-29 | 状态: Sprint E1 规划中 | 依赖: M30-PRD-实时消息系统, P27-PRD-消息中心
> 用户端入口: 商家/圣地/祖庭详情页 "联系商家" 按钮
> 商家端入口: Admin后台 → 咨询管理

---

## 1. 需求背景与目标

### 1.1 背景

商家咨询是连接朝圣者与服务提供方(祖庭、圣地运营商、导游服务商)的关键沟通通道。用户在浏览圣地详情、祖庭详情时，常有关于行程安排、住宿素斋、交通接送、开放时间等具体问题，需要直接与商家沟通。

当前缺乏平台内咨询入口，用户需通过电话或微信等外部渠道联系商家，导致:
- 咨询记录无法追踪，服务质量无法衡量
- 用户从浏览到咨询的链路断裂，转化率低
- 商家无法统一管理来自不同渠道的咨询

参考 Booking.com 酒店详情页的"联系住宿方"按钮和 Airbnb 房源页的"联系房东"功能，在JOINUS详情页提供一键咨询入口，创建 MERCHANT_INQUIRY 类型聊天室，并在Admin后台为商家提供咨询管理界面。

### 1.2 目标

| 目标 | 指标 | 说明 |
|------|------|------|
| 咨询发起率 | >8% | 详情页访客中发起咨询的比例 |
| 咨询→下单转化 | >15% | 发起咨询后最终创建行程/下单的用户 |
| 商家首次响应 | <5min | 商家收到咨询后的平均首次回复时长 |
| 咨询解决率 | >90% | 用户咨询后得到有效回复的比例 |

### 1.3 范围

| 范围 | v1 (Sprint E1) | v2 |
|------|----------------|-----|
| 用户端入口 | 详情页"联系商家"按钮 | 浮窗气泡入口+智能问候语 |
| 聊天室创建 | 创建MERCHANT_INQUIRY聊天室 | 自动匹配在线客服 |
| 商家端管理 | Admin后台查看+回复消息 | 快捷回复模板+自动回复 |
| 商家端统计 | — | 咨询量/响应时长/满意度仪表盘 |
| 智能分流 | — | 小鸿AI先应答常见问题 |

## 2. 用户故事

### US-28-01: 用户从详情页发起咨询

**作为** 朝圣者，**我希望** 在圣地/祖庭详情页直接联系商家，**以便** 了解行程细节和服务内容。

**验收标准:**
- 圣地详情页、祖庭详情页底部固定栏显示"联系商家"按钮
- 按钮样式: 金色描边 (#D4A855)，图标为聊天气泡，文字"咨询商家"
- 点击按钮:
  - 若用户未登录 → 跳转登录页，登录后回到详情页
  - 若已有与该商家的活跃聊天室 → 直接跳转到消息中心对应聊天室
  - 若无活跃聊天室 → 创建 MERCHANT_INQUIRY 聊天室，自动发送首条系统消息
- 首条系统消息内容: "您正在咨询 [商家名称] — [圣地/祖庭名称]"，附带详情页链接卡片

### US-28-02: 用户填写咨询内容

**作为** 朝圣者，**我希望** 在发起咨询时能说明我的问题，**以便** 商家快速理解我的需求。

**验收标准:**
- 点击"咨询商家"后，弹出快捷咨询面板 (底部弹窗 / 对话框)
- 面板包含:
  - 常见问题快捷标签: "开放时间"、"住宿素斋"、"交通接送"、"门票价格"、"行程安排"、"其他"
  - 自定义输入框: "请描述您的问题..."
- 选择标签或输入文字后点击"发送咨询"
- 创建聊天室 + 发送用户第一条消息
- 跳转到消息中心对应聊天窗口

### US-28-03: 商家在Admin后台回复咨询

**作为** 商家/管理员，**我希望** 在后台看到用户咨询并回复，**以便** 及时解答用户疑问促成下单。

**验收标准:**
- Admin后台左侧菜单新增"咨询管理"入口 (聊天气泡图标)
- 咨询管理页面布局:
  - 左侧: 咨询列表 (用户头像+名字+最后消息+时间+未回复标记)
  - 右侧: 聊天窗口 (与用户端消息中心相同的气泡+输入框)
- 咨询列表支持按状态筛选: 全部 / 未回复 / 已回复 / 已关闭
- 未回复的咨询在菜单入口显示红色Badge数字
- 商家可发送TEXT和IMAGE消息
- 商家可查看用户咨询来源(哪个圣地/祖庭详情页)

### US-28-04: 咨询关联行程

**作为** 商家，**我希望** 在咨询中看到用户是否有关联行程，**以便** 提供针对性的服务建议。

**验收标准:**
- 聊天窗口右侧面板(或顶部折叠区)显示:
  - 用户基本信息: 头像、昵称
  - 咨询来源: 圣地/祖庭名称 (可点击跳转)
  - 关联行程: 行程标题+状态 (若有)
  - 历史咨询次数
- 无关联行程时显示"暂无行程"

### US-28-05: 用户收到商家回复通知

**作为** 朝圣者，**我希望** 商家回复后及时收到通知，**以便** 不错过重要信息。

**验收标准:**
- Web端: 导航栏消息图标未读数更新 + 浏览器标签标题 "[N] 消息"
- Mobile端: 底部Tab消息Badge + 推送通知 (v2)
- 小程序端: TabBar消息Badge + 模板消息 (v2)
- 进入消息中心后商家聊天室排在最前 (有新消息)

## 6. API调用映射

### 6.1 用户端

| 用户操作 | API | 说明 |
|----------|-----|------|
| 点击"咨询商家" | `GET /api/chat/rooms?type=MERCHANT_INQUIRY&merchantId=xxx` | 查询已有聊天室 |
| 创建咨询 | `POST /api/chat/rooms` body: `{type:'MERCHANT_INQUIRY', merchantId, name, ...}` | 创建聊天室 |
| 发送首条消息 | `ws:send_message` | 用户第一条咨询消息 |
| 后续消息 | 复用P27消息中心全部交互 | — |

### 6.2 商家端 (Admin)

| 操作 | API | 说明 |
|------|-----|------|
| 加载咨询列表 | `GET /api/chat/rooms?type=MERCHANT_INQUIRY&role=MERCHANT` | 按商家身份筛选 |
| 查看消息 | `GET /api/chat/rooms/:id/messages` | 加载聊天记录 |
| 回复消息 | `ws:send_message` | Admin端也走WebSocket |
| 标记已读 | `POST /api/chat/rooms/:id/read` | — |
| 查看未回复数 | `GET /api/chat/unread-count?type=MERCHANT_INQUIRY` | 菜单Badge |

## 8. 错误处理

| 场景 | 处理方式 |
|------|---------|
| 未登录点击咨询 | Toast提示"请先登录"，跳转登录页，登录后回到详情页 |
| 商家不存在 | 按钮不显示 (前端根据merchantId判断) |
| 创建聊天室失败 | Toast提示"创建失败，请稍后重试" |
| 商家长时间未回复 | 24h未回复时发送系统消息"商家暂时繁忙，请耐心等待或拨打客服电话" |
| Admin WebSocket断开 | 顶部Banner提示"连接断开"，自动重连，新消息在重连后补拉 |
