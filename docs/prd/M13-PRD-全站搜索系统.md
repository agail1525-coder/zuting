## 1. 背景与目标

### 1.1 背景

JOINUS.COM 当前拥有 12 大信仰、60 处圣地、27 祖庭、28 祖师、39 祖训等丰富内容资产，但用户只能通过逐级导航菜单进行被动浏览，缺乏主动搜索入口。全站无统一搜索功能，导致用户在寻找特定圣地或祖庭时严重依赖分类导航，探索效率低、转化路径长。对标 Booking.com、Kayak、Skyscanner 等头部平台，搜索框是用户进入内容的第一路径，是平台留存与转化的核心入口。

### 1.2 目标

- [ ] 用户在全站任意页面均可发起搜索，搜索覆盖圣地/祖庭/祖师/祖训 4 类实体
- [ ] 搜索建议（Autocomplete）响应时间 ≤ 200ms（P95）
- [ ] 搜索结果支持宗教/类型/地区 3 维筛选，支持相关度/热门/名称/距离 4 种排序
- [ ] 搜索系统上线后，用户内容发现路径缩短，圣地详情页 PV 提升 ≥ 20%
- [ ] 移动端（Expo App + 微信小程序）搜索体验与 Web 功能对等

### 1.3 非目标（本期不做的）

- 不包含商品/服务价格搜索（Phase C 支付模块负责）
- 不包含用户账号/行程/订单的站内搜索（仅限内容实体）
- 不包含语音搜索（v2 延期）
- 不包含基于行为数据的语义搜索/向量检索（Phase D AI 增强搜索）
- 不包含搜索结果广告位投放（v3）

## 2. 竞品对标分析

### 2.1 竞品功能对比

| 竞品 | 核心功能 | 优点 | 缺点 | 借鉴点 |
|------|---------|------|------|-------|
| Booking.com | 目的地搜索框+日期+人数一体化，搜索结果含地图/列表双视图，左侧筛选栏可多选 | 筛选器功能极其丰富，支持50+维度；结果即时响应；地图与列表无缝切换 | 筛选项太多导致信息过载，新用户困惑 | 左侧筛选栏+主内容区双栏布局；地图/列表切换按钮置顶 |
| Kayak | 多维度筛选器（价格区间/时间/评分/设施），价格日历，支持排序切换 | 筛选器设计精简，信息密度高但不乱；价格区间滑块易用 | 过度聚焦机票/酒店，文化内容场景弱 | 横向顶部筛选栏（Tab式）+排序下拉；滑块式区间筛选 |
| Skyscanner | 极简搜索框+弹出日历，结果页有"最便宜日期"热力图 | UI极简，聚焦主要任务；搜索框交互动画流畅；移动端体验优秀 | 功能垂直于机票，通用搜索能力弱 | 极简搜索框设计；输入时全屏遮罩聚焦；搜索建议分组显示 |
| Trip.com | 首页大搜索框+Tab切换（酒店/机票/火车），搜索联想词高亮显示 | 搜索建议分类清晰（目的地/酒店/景点分组）；中文分词效果好；移动端手势体验佳 | 内容与商业产品耦合重，纯内容搜索体验次 | 搜索建议分类分组展示；中文关键词高亮；搜索历史置顶展示 |
| TripAdvisor | 全局搜索覆盖景点/餐厅/酒店，结果卡片含评分+评价数 | UGC评分增强结果可信度；搜索词自动关联周边 | 搜索算法有时结果不精准 | 结果卡片集成评分展示；搜索结果可按评分排序 |

### 2.2 差距分析

当前 JOINUS.COM 缺少统一搜索入口，用户无法跨类型（圣地/祖庭/祖师）搜索，也无法通过关键词快速定位特定内容。这导致：
- 用户知道目标（如"法门寺"）但无法快速找到，只能逐级点击导航
- 移动端用户体验尤为糟糕，4-5级导航路径在手机上操作繁琐
- 平台内容资产（196条记录）利用率低，部分冷门但高价值内容几乎无法被发现

### 2.3 差异化优势

JOINUS.COM 的搜索在通用旅行搜索之外，具备宗教文化垂直深度：
- 搜索结果按信仰体系分组，帮助朝圣者在信仰维度内探索
- 圣地搜索结果展示信仰符号和圣地类型（朝圣地/祖庭/修行地），这是通用旅行平台不具备的维度
- 支持宗教典籍关键词（祖训原文片段）的全文检索，满足学术型用户需求

```
用户点击搜索图标 → 展开搜索遮罩 → 显示历史+热门
  ↓

  ↓
下拉建议展示（分组：圣地/祖庭/其他） → 用户选择或回车
  ↓

  ↓
结果页展示骨架屏 → API返回 → 渲染结果卡片
```

## 6. API 契约

### 6.1 API 端点总览

| 端点 | 方法 | 用途 | 认证 |
|------|------|------|------|
| `/api/search` | GET | 全文搜索，返回分页结果 | 公开 |
| `/api/search/suggestions` | GET | 搜索建议（Autocomplete） | 公开 |
| `/api/search/hot` | GET | 热门搜索词TOP10 | 公开 |
| `/api/search/map` | GET | 地图搜索，返回标记点数组 | 公开 |
| `/api/search/history` | GET | 获取用户搜索历史 | JWT必须 |
| `/api/search/history` | POST | 记录搜索词到历史 | JWT必须 |
| `/api/search/history` | DELETE | 清空搜索历史 | JWT必须 |

### 6.2 主搜索接口

#### `GET /api/search`

**请求参数:**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `q` | string | 是 | 搜索关键词，最少2字符，最多100字符 |
| `type` | string | 否 | 实体类型：all/holy-site/temple/patriarch/teaching，默认 all |
| `religion` | string | 否 | 信仰slug，多值用逗号分隔，如 buddhism,taoism |
| `country` | string | 否 | 国家代码，如 CN/JP/SA |
| `sort` | string | 否 | 排序：relevance/popular/name/distance，默认 relevance |
| `lat` | number | 否 | 用户纬度（sort=distance时必填） |
| `lng` | number | 否 | 用户经度（sort=distance时必填） |
| `page` | number | 否 | 页码，默认1 |
| `pageSize` | number | 否 | 每页数量，默认20，最大50 |

**响应格式:**

```json
{
  "items": [
    {
      "id": "clxxx",
      "type": "holy-site",
      "name": "少林寺",
      "nameEn": "Shaolin Temple",
      "religion": { "slug": "buddhism", "nameZh": "佛教", "color": "#FF6B35" },
      "country": "CN",
      "description": "禅宗祖庭，武术圣地...",
      "imageUrl": "https://...",
      "lat": 34.5,
      "lng": 112.9,
      "score": 0.85
    }
  ],
  "total": 8,
  "page": 1,
  "pageSize": 20,
  "took": 45
}
```

**错误码:**

| HTTP状态码 | 错误码 | 说明 |
|------------|-------|------|
| 400 | QUERY_TOO_SHORT | 关键词少于2字符 |
| 400 | INVALID_TYPE | 非法实体类型值 |
| 429 | RATE_LIMIT | 搜索频率超限（10次/分钟/IP） |
| 500 | SEARCH_ERROR | 搜索引擎内部错误 |

### 6.3 搜索建议接口

#### `GET /api/search/suggestions`

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `q` | string | 是 | 前缀关键词，最少2字符 |
| `limit` | number | 否 | 返回数量，默认8，最大15 |

**响应格式:**

```json
{
  "suggestions": [
    { "type": "holy-site", "id": "clxxx", "name": "少林寺", "imageUrl": "...", "religion": "佛教" },
    { "type": "keyword", "text": "少林寺武术" },
    { "type": "temple", "id": "clyyy", "name": "少林禅寺", "imageUrl": "...", "religion": "佛教" }
  ]
}
```

## 7. 数据模型

### 7.1 新增 Prisma 模型

```prisma
// 热门搜索词统计（持久化到PostgreSQL，供冷启动）
model SearchStat {
  id        String   @id @default(cuid())
  keyword   String   @unique
  count     Int      @default(0)
  updatedAt DateTime @updatedAt
}
```

### 7.2 现有表的索引扩展（PostgreSQL原生SQL，不修改Prisma Schema）

```sql
-- 启用扩展
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS unaccent;

-- HolySite 全文检索索引
ALTER TABLE "HolySite" ADD COLUMN IF NOT EXISTS search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple',
      coalesce(name, '') || ' ' || coalesce("nameZh", '') || ' ' ||
      coalesce(description, '') || ' ' || coalesce("descriptionZh", '') || ' ' ||
      coalesce(location, '') || ' ' || coalesce(country, '')
    )
  ) STORED;
CREATE INDEX IF NOT EXISTS idx_holysite_fts ON "HolySite" USING GIN(search_vector);
CREATE INDEX IF NOT EXISTS idx_holysite_trgm ON "HolySite" USING GIN(name gin_trgm_ops);

-- Temple / Patriarch / Teaching 同上（略）
```

### 7.3 Redis 数据结构

| Key | 类型 | 说明 | TTL |
|-----|------|------|-----|
| `search:hot` | SortedSet | 全局热搜词，score为搜索次数 | 无（定期增量更新） |
| `search:history:{userId}` | List | 用户搜索历史，最多20条 | 30天 |
| `search:suggestions:{hash}` | String | 建议词缓存 | 5分钟 |
| `search:result:{hash}` | String | 搜索结果缓存 | 2分钟 |

## 8. 技术约束

- **搜索引擎**: PostgreSQL FTS（pg_trgm + tsvector + GIN索引），无需引入Elasticsearch
- **中文分词**: 使用 `simple` 配置（字符分割），配合 pg_trgm 实现中文模糊匹配，满足MVP需求；v2升级zhparser
- **性能要求**: 搜索建议 P95 ≤ 200ms；全文搜索 P95 ≤ 500ms；地图搜索 P95 ≤ 300ms
- **移动端**: 必须在 375px 宽度下正常显示，全屏搜索模式适配软键盘
- **认证**: 搜索/建议/热搜为公开接口；搜索历史接口需 JWT 认证（@Public() 不标记）
- **限流**: 搜索接口 10次/分钟/IP，建议接口 30次/分钟/IP（防止爬取）
- **新增依赖**: 无需引入新npm包，使用已有 Prisma + NestJS 能力
