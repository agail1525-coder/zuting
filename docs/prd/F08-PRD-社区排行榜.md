## 1. 背景与目标

### 1.1 业务背景

社区排行榜是激励用户持续产出内容、提升平台活跃度的核心机制。
当前平台已有游记(journals)、评价(reviews)、行程(trips)三类用户内容，但缺乏公开的荣誉榜来激励贡献。

用户痛点:
- 高质量内容贡献者得不到可见的社区认可
- 用户之间缺乏良性竞争驱动力
- 新用户不知道社区里谁是"大神"，无法快速找到高质量内容

### 1.2 目标

- 通过公开排行榜激励 UGC 内容产出
- 让优质内容贡献者获得社区认可，强化身份认同
- 为新用户提供"寻找达人"的入口，引导内容消费
- 与朝圣等级体系协同，形成完整的社区荣誉体系

## 5. 榜单计分规则

### 5.1 游记达人榜

```
分数 = 发布游记数 × 1
     + 游记获赞数 × 0.5
     + 游记被收藏数 × 1
     + 游记评论数 × 0.3

周榜: 统计本自然周 (周一0点 ~ 周日23:59) 内的增量
月榜: 统计本自然月 (1号0点 ~ 最后一天23:59) 内的增量
总榜: 统计账号历史累计数据
```

### 5.2 评价达人榜

```
分数 = 发布评价数 × 1
     + 评价获赞数 × 0.5
     + 评价被标记"有帮助"数 × 1
     + 评价含图片数 × 0.3 (上限5分/条)

排名逻辑同游记达人榜
```

### 5.3 朝圣达人榜

```
分数 = 到访圣地总数 × 5
     + 完成行程数 × 3
     + 朝圣等级 × 10
     + 游记数 × 1
     + 评价数 × 0.5

朝圣达人榜只有"总榜"维度(无周/月，因为到访圣地是累计性数据)
```

### 5.4 趋势内容计算 (热门游记/评价)

```
趋势分 = 过去7天内:
  浏览数 × 0.1
  + 点赞数 × 3
  + 收藏数 × 5
  + 评论数 × 2
  + 分享数 × 4

取趋势分最高的前6条内容 (游记3 + 评价3)
```

## 7. API 契约

### 7.1 GET /api/community/leaderboard

**描述**: 获取指定类型和时间维度的排行榜 TOP 10

**请求**:
```
GET /api/community/leaderboard?type=guides&period=week
Authorization: Bearer <token>  (可选，登录后额外返回 myRank)
```

**Query 参数**:

| 参数 | 类型 | 必填 | 默认值 | 可选值 |

## 8. 后端计算策略

### 8.1 榜单刷新机制

| 榜单类型 | 刷新时间 | 计算方式 |
|---------|---------|---------|
| 周榜 | 每周一 00:05 定时任务 | 统计上周增量数据，写入缓存 |
| 月榜 | 每月1日 00:05 定时任务 | 统计上月增量数据，写入缓存 |
| 总榜 | 每日 02:00 定时任务 | 全量重算，写入缓存 |
| 趋势内容 | 每小时 :00 定时任务 | 滚动7天窗口，写入缓存 |

### 8.2 缓存 Key 设计 (Redis)

```
leaderboard:guides:week        → 游记达人周榜 TOP10 (TTL: 到下次刷新)
leaderboard:guides:month       → 游记达人月榜 TOP10
leaderboard:guides:all         → 游记达人总榜 TOP10
leaderboard:reviews:week       → 评价达人周榜 TOP10
leaderboard:reviews:month      → 评价达人月榜 TOP10
leaderboard:reviews:all        → 评价达人总榜 TOP10
leaderboard:trips:all          → 朝圣达人总榜 TOP10
trending:content               → 热门内容 (TTL: 600s)
```

### 8.3 用户排名实时查询

`myRank` 数据不预缓存，每次请求时通过以下SQL实时查询:

```sql
-- 示例: 查询当前用户在游记达人周榜的排名
WITH ranked AS (
  SELECT
    up.userId,
    up.guideCount AS count,
    RANK() OVER (ORDER BY up.guideCount DESC) AS rank
  FROM UserProfile up
)
SELECT rank, count FROM ranked WHERE userId = :userId;
```

> 注: 实际周榜需改为查询本周增量游记数，此为示意SQL。

// 热门内容查询

  queryKey: ['trending'],

| 状态 | 处理方式 |
|------|---------|
| 榜单数据为空 | "本周还没有内容，快来第一个上榜！" + [去写游记] 按钮 |
| API请求失败 | 显示错误提示 + 重试按钮，不显示空白区域 |
| 未登录访问"我的排名" | 显示"登录后查看你的排名" + [登录] 按钮 |
| 加载中 | 榜单项用骨架屏(Skeleton)占位 |
| 热门内容为空 | 隐藏趋势内容区域，不显示空块 |

---

## 11. 性能要求

- 榜单页面首屏 LCP < 2s
- 榜单 API 响应时间 < 200ms (依赖 Redis 缓存)
- 趋势内容 API 响应时间 < 100ms
- 榜单数据前端缓存 5 分钟 (staleTime)，减少重复请求
- 热门内容图片使用懒加载 + WebP 格式

---

## 12. 安全要求

- 排行榜 API 为公开接口，无需 JWT
- `myRank` 字段通过 JWT 解析 userId，不接受客户端传入 userId (防 IDOR)
- 榜单数据只读，不提供写入接口
- 限流: 同 IP 每分钟最多 120 次请求
