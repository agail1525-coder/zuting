## 1. 背景与目标

### 1.1 背景

搜索建议下拉框（Autocomplete Dropdown）是搜索体验的第一个交互节点。用户在输入过程中即获得建议，可以减少完整输入的心理负担，同时引导用户发现平台内容。对标 Trip.com 的分组建议、Google 的关键词高亮、Booking.com 的缩略图+目的地名称展示，JOINUS.COM 的搜索建议需要在简洁的下拉框内展示分类建议和实体卡片，提升搜索意图匹配效率。

### 1.2 目标

- [ ] 输入 ≥ 2 字符后 200ms 内展示搜索建议（防抖后触发）
- [ ] 建议按类型分组（实体匹配 / 热门关键词），每组最多5条
- [ ] 键盘上下键可导航建议列表，Enter确认，Esc关闭
- [ ] Mobile端进入全屏搜索模式，建议列表不被软键盘遮挡
- [ ] 微信小程序用Taro原生组件实现兼容版本

### 1.3 非目标（本期不做的）

- 不包含语音输入建议（v2）
- 不包含多语言（英/日/韩）关键词联想（Phase D）
- 不包含基于用户行为的个性化建议排序（Phase D）

## 2. 竞品对标分析

### 2.1 竞品功能对比

| 竞品 | 建议内容 | 交互特点 | 借鉴点 |
|------|---------|---------|-------|
| Trip.com | 目的地名称+缩略图+国家，分"目的地/酒店/景点"分组 | 输入即时触发，中文分词效果佳，建议词高亮匹配部分 | 分组标题+分组内容列表；建议词高亮；每条带缩略图 |
| Booking.com | 目的地+小地图缩略图，类型标签（城市/酒店/地区） | 第一条默认高亮；键盘可导航；鼠标hover高亮 | 第一条自动高亮；类型标签（圣地/祖庭/...） |
| Google搜索 | 关键词补全+相关搜索+问题形式 | 输入即时，建议词加粗显示用户已输入部分 | 已输入部分正常色，补全部分加粗高亮 |
| Skyscanner | 机场IATA+城市名，搜索时全屏遮罩 | 打开时背景遮罩，焦点明确；建议5-6条 | 全屏搜索遮罩设计；建议条数精简 |

### 2.2 差距分析

JOINUS.COM 当前各列表页均无 Autocomplete 功能，用户只能通过完整搜索词跳转结果页，缺少"边输入边发现"的体验。这导致输入成本高，拼写不确定时无从参考。

### 2.3 差异化优势

建议条目展示信仰图标（emoji或SVG图标）+信仰名称，帮助用户在选择建议时即可识别该内容的宗教归属，这是旅游平台不具备的宗教文化维度。

## 4. 功能清单

### 4.1 核心功能（P0 — 必须有）

- [ ] 输入防抖：输入停止后 200ms 触发 API 请求
- [ ] 触发条件：输入字符数 ≥ 2 时触发；< 2 字符时显示历史+热门
- [ ] 分组展示：建议按"匹配内容"和"热门关键词"两组展示
- [ ] 实体建议条目：含缩略图（50x50px）、中文名称、类型标签（圣地/祖庭/...）、信仰名
- [ ] 关键词建议条目：仅文字，无图片，灰色搜索图标前缀
- [ ] 已输入部分高亮：建议文本中与输入词匹配的部分用金色高亮
- [ ] 点击建议条目：实体类型直接跳转详情页；关键词类型跳转搜索结果页
- [ ] 关闭逻辑：点击下拉框外区域关闭；Esc键关闭；跳转后关闭
- [ ] Mobile全屏模式：搜索框获得焦点时展开全屏搜索界面

### 4.2 增强功能（P1 — 应该有）

- [ ] 键盘导航：上下键移动高亮项，Enter确认，Home/End跳到首尾
- [ ] 无结果建议：API返回空时，显示"尝试搜索..."提示 + 当前输入词
- [ ] 请求取消：新请求发出时，取消前一个未完成的请求（AbortController）
- [ ] 网络错误降级：建议API失败时静默隐藏下拉，不阻断输入体验
- [ ] 小程序适配：使用Taro的`<Input>`+自定义建议列表实现

### 4.3 优化功能（P2 — 可以有）

- [ ] 建议词缓存：相同前缀的建议结果缓存5分钟（前端内存缓存）
- [ ] 骨架屏建议：请求中显示2-3条骨架建议（灰色占位）
- [ ] 最近访问高优先：用户最近浏览的实体出现在建议前排（本地localStorage）

## 5. 页面线框 / 交互设计

### 5.1 下拉框结构

```
+--------------------------------------------------+
|  🔍 [少林                               ] [×]    |
+--------------------------------------------------+
|  匹配内容                                         |
|  ┌──────────────────────────────────────────┐    |
|  │ [🖼] 少林寺  ←高亮   圣地 · 佛教            │    |
|  │ [🖼] 少林禅寺         祖庭 · 佛教            │    |
|  │ [🖼] 少林惠能纪念堂   圣地 · 佛教            │    |
|  └──────────────────────────────────────────┘    |
|  热门关键词                                       |
|  ┌──────────────────────────────────────────┐    |
|  │ 🔍 少林寺武术表演                           │    |
|  │ 🔍 少林禅修体验                             │    |
|  └──────────────────────────────────────────┘    |
+--------------------------------------------------+
```

### 5.2 Mobile 全屏搜索模式

```
+--------------------------------+
|  ← 返回     [少林          ] ×  |   ← 粘性顶部，不被键盘顶走
+================================+ ← 软键盘上方区域
|  匹配内容                       |
|  [🖼] 少林寺     圣地·佛教      |
|  [🖼] 少林禅寺   祖庭·佛教      |
|                                 |
|  热门关键词                      |
|  🔍 少林寺武术                  |
|  🔍 少林禅修                    |
+--------------------------------+
|  [                            ] | ← 软键盘（系统弹出）
+--------------------------------+
```

### 5.3 关键交互流程

```
用户在搜索框输入
  ↓
字符数 < 2
  → 展示：最近搜索历史（顶部）+ 热门搜索（下方）

字符数 ≥ 2
  → 200ms防抖计时
  → 触发 GET /api/search/suggestions?q={input}
  → 显示加载动画（可选骨架屏）
  → 渲染分组建议

用户键盘上下键导航
  → 高亮项上下移动（环形：到底后回到顶部）
  → 高亮时搜索框预填充该建议词（可选）

用户按 Enter
  → 如有高亮项：执行该项的跳转行为
  → 无高亮项：以当前输入词跳转搜索结果页

用户按 Esc
  → 关闭下拉，搜索框保留输入内容，失去焦点
```

### 5.4 三态设计

| 状态 | 触发条件 | 表现 |
|------|---------|------|
| Loading | 请求进行中（输入防抖后到响应前） | 下拉框显示，条目位置显示3条骨架（可选，时间短可跳过） |
| Empty（建议为空） | API返回空建议数组 | 下拉框内显示 "没有找到"{input}"的建议，按Enter搜索全站" |
| Error（网络失败） | API请求超时或失败 | 静默：隐藏下拉框，不显示错误（用户可继续输入或回车搜索） |

## 6. API 契约

### 6.1 使用的 API 端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `GET /api/search/suggestions` | GET | 获取搜索建议列表 |
| `GET /api/search/hot` | GET | 输入< 2字符时展示热门搜索 |
| `GET /api/search/history` | GET | 输入< 2字符时展示搜索历史 |

### 6.2 建议接口详情

#### `GET /api/search/suggestions`

**请求参数:**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `q` | string | 是 | 输入前缀，2-50字符 |
| `limit` | number | 否 | 返回建议总数，默认8，最大15 |

**响应格式:**

```json
{
  "suggestions": [
    {
      "group": "entity",
      "type": "holy-site",
      "id": "clxxx",
      "name": "少林寺",
      "nameEn": "Shaolin Temple",
      "imageUrl": "https://cdn.joinus.com/holy-sites/shaolin.jpg",
      "religion": "佛教",
      "religionColor": "#FF6B35"
    },
    {
      "group": "keyword",
      "text": "少林寺武术表演"
    }
  ],
  "took": 45
}
```

**建议条目类型说明:**

| group | 字段 | 跳转行为 |
|-------|------|---------|
| `entity` | id + type + name + imageUrl + religion | 按type跳转详情页 |
| `keyword` | text | 跳转 /search?q={text} |

## 7. 数据模型

本功能不新增数据模型，复用 M13 主PRD 中的搜索索引和Redis缓存。

建议内容来源：
- `entity` 建议：从 PostgreSQL FTS 索引查询，取relevance排名前5
- `keyword` 建议：从 Redis `search:hot` SortedSet 中前缀匹配

## 8. 技术约束

- **防抖**: 使用 `useDebounce(value, 200)` 控制请求频率
- **请求取消**: 每次新请求使用 `AbortController`，在新请求触发时取消前一个
- **Web组件**: 封装为 `<SearchAutocomplete>` 独立组件，供 Header 和搜索结果页复用
- **Mobile**: Expo环境下使用 `KeyboardAvoidingView` 确保建议列表不被软键盘遮挡
- **小程序**: Taro环境中使用 `Input` 组件的 `onInput` 事件，自定义 `View` 实现建议列表
- **可访问性**: 下拉框使用 `role="listbox"`，建议条目使用 `role="option"`，支持 ARIA 属性
- **类型安全**: 建议条目使用 `SearchSuggestionItem` 联合类型（EntitySuggestion | KeywordSuggestion）

| 编号 | 验收项 | 测试步骤 | 通过标准 |
