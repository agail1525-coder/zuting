## 1. 页面定位

支付结果页是用户完成支付动作后的落地页，承担三个核心职责：
1. **及时反馈**：明确告知用户支付成功/失败/处理中
2. **情感收尾**：成功时强化"朝圣之旅即将开始"的仪式感
3. **引导下一步**：减少用户离开后的困惑，引导至订单管理或继续探索

### 1.1 竞品对标

| 竞品 | 成功页设计 | 失败页设计 |
|------|-----------|-----------|
| Booking.com | 绿色大勾 + 订单号 + 邮件提示 + 推荐类似酒店 | 明确错误原因 + 重试按钮 |
| Agoda | 动效勾（Lottie） + 积分获得提示 + 分享按钮 | 错误码 + 客服入口 |
| 飞猪 | 大红"预订成功" + 行程时间提示 + 添加日历 | 红色叉 + 常见问题解答 |
| Airbnb | 预订确认界面 + 地图（目的地） + 联系房东入口 | 简洁文案 + 重试 |

```

US-06: 作为用户，成功后我希望继续浏览其他圣地，探索更多可能
```

## 5. API 契约

### 5.1 查询订单支付状态

```
GET /api/orders/:orderId/payment-status
Authorization: Bearer {token}
```

**Response 200:**
```json
{
  "orderId": "order_yyy",
  "status": "PAID | PENDING | PAYMENT_FAILED | CANCELLED",
  "paymentAmount": 3427.00,
  "paymentMethod": "WECHAT_PAY | ALIPAY | STRIPE",
  "paymentAt": "2026-05-01T10:32:18Z",
  "failureReason": null,
  "failureCode": null,
  "trip": {
    "title": "丝绸之路朝圣7日游",
    "startDate": "2026-05-01",
    "endDate": "2026-05-07",
    "participants": 2
  }
}
```

**失败时 failureCode 枚举 → 前端映射为中文提示:**
```
INSUFFICIENT_BALANCE   → "银行卡余额不足"
CARD_DECLINED          → "银行卡被拒绝，请联系发卡行"
NETWORK_TIMEOUT        → "网络超时，支付未完成"
RISK_CONTROL           → "支付被风控拦截，请更换支付方式"
ORDER_EXPIRED          → "订单已过期，请重新下单"
USER_CANCELLED         → "用户主动取消支付"
UNKNOWN                → "支付失败，请联系客服"
```

### 5.2 取消订单

```
POST /api/orders/:orderId/cancel
Authorization: Bearer {token}
```

**Request Body:**
```json
{
  "reason": "支付失败后放弃" // 可选
}
```

**Response 200:**
```json
{ "orderId": "order_yyy", "status": "CANCELLED" }
```

### 5.3 重新发起支付

```
POST /api/orders/:orderId/pay
Authorization: Bearer {token}
```

**Request Body:**
```json
{
  "paymentMethod": "ALIPAY"  // 可选，切换支付方式
}
```

**Response 200:**
```json
{
  "paymentUrl": "alipays://...",
  "qrCodeUrl": "https://...",
  "expiredAt": "2026-05-01T10:47:00Z"
}
```

## 6. 轮询逻辑

### 6.1 PROCESSING 状态下的轮询

```typescript
// 伪代码
const MAX_POLL = 20;
const POLL_INTERVAL = 3000; // 3秒

let count = 0;
const poll = async () => {
  if (count >= MAX_POLL) {
    showManualCheckButton(); // 展示"查看订单状态"按钮
    return;
  }
  const result = await fetchPaymentStatus(orderId);
  if (result.status === 'PAID') {
    showSuccess(result);
  } else if (result.status === 'PAYMENT_FAILED' || result.status === 'CANCELLED') {
    showFailure(result);
  } else {
    count++;
    setTimeout(poll, POLL_INTERVAL);
  }
};
poll();
```

### 6.2 页面进入逻辑

```
URL 格式: /payment/result?orderId={orderId}

进入时:
1. 立即调用 GET /api/orders/:orderId/payment-status
2. PAID → 展示成功页
3. PAYMENT_FAILED / CANCELLED → 展示失败页
4. 其他状态（PENDING / PAYING）→ 展示处理中页，启动轮询
```

## 7. 状态与错误处理

| 场景 | 处理方式 |
|------|---------|
| orderId 不存在 | 展示错误页："订单不存在，请返回首页" |
| 无权限查看该订单（他人订单） | 403 → 重定向登录页 |
| 网络请求失败 | Toast + 重试按钮，不展示具体状态 |
| 轮询20次仍 PENDING | 提示"请稍后在'我的订单'中查看结果" |
| 用户刷新成功页 | 重新请求 API，正常展示（不依赖页面状态） |

## 8. 技术约束

```
Web路由:  /payment/result?orderId=xxx
Mobile:   /payment/result/[orderId]

[R-63]   GET /api/orders/:id/payment-status 必须有 JwtAuthGuard
[R-68]   校验 order.userId === 当前用户 ID，防止 IDOR
[B4]     enabled: !!orderId — orderId 为空时不发请求
[HH-V03] 必须真实轮询 API，禁止 setTimeout 模拟状态切换

动效方案(Mobile优先):
  - 成功: react-native-reanimated checkmark 动效
  - 处理中: ActivityIndicator 或 Lottie
  - Web: CSS keyframes 或 lottie-web
  - 依赖动效库为 P1，CSS 纯色图标为 P0 兜底
```
