## 1. 背景与目标

### 1.1 背景

UGC 内容的健康生态依赖人工审核机制。评价在提交后进入 PENDING 状态，管理员需要在 Admin 后台及时审核、处理违规内容并进行官方回复，以维护平台内容质量和社区信任度。TripAdvisor 和 Booking.com 均有成熟的内容审核体系，JOINUS.COM 作为宗教文化平台尤其需要对敏感内容进行严格把关。

### 1.2 目标

- [ ] Admin 后台新增"评价管理"页面，展示待审核/已通过/已拒绝/已隐藏评价列表
- [ ] 支持单条审核（通过/拒绝）、官方回复、隐藏等操作，以及批量审核
- [ ] Dashboard 首页新增"待审核评价"数量 Badge 作为快速入口
- [ ] 提供评价统计图表（日评价量趋势）辅助运营决策

### 1.3 非目标（本期不做的）

- 不包含 AI 自动审核（延期到 Phase D）
- 不包含导出 CSV 报告（延期到 Phase C）
- 不包含评价来源渠道分析（延期到 Phase D 数据分析模块）

## 2. 竞品对标分析

### 2.1 竞品功能对比

| 竞品 | 功能描述 | 优点 | 缺点 | 借鉴点 |
|------|---------|------|------|-------|
| TripAdvisor 管理中心 | 分类筛选（星级/时间/状态），官方回复内嵌，批量删除 | 操作路径短，官方回复入口直接 | 批量操作功能有限 | 审核列表布局、官方回复内联编辑 |
| Booking.com 合作伙伴管理 | 未回复评价优先显示，回复有字数要求和模板 | 回复模板减少工作量 | 功能过于针对商家 | 未处理评价优先显示（PENDING 排前）|
| 大众点评商家后台 | 评价筛选+回复+申诉+统计图 | 功能全面，统计维度丰富 | 界面较复杂 | 日评价量趋势图、按星级筛选 |

### 2.2 差距分析

Admin 后台当前无评价管理功能，管理员无法履行内容审核职责，所有 UGC 评价提交后无法控制，存在合规风险。

### 2.3 差异化优势

JOINUS.COM 评价管理增加"宗教敏感内容"快速标记，针对宗教冲突言论、不当言论等宗教文化垂直场景的合规需求提供专项处理流程。

## 4. 功能清单

### 4.1 核心功能（P0 — 必须有）

- [ ] 评价审核列表页：展示所有评价，支持按状态筛选（待审核/已通过/已拒绝/已隐藏）
- [ ] 审核通过操作：一键将 PENDING → APPROVED，同步更新 HolySite/Temple.avgRating
- [ ] 审核拒绝操作：PENDING → REJECTED，必填拒绝理由，触发通知给用户
- [ ] 隐藏评价操作：APPROVED → HIDDEN（已通过违规的紧急处理）
- [ ] 分页列表：每页 20 条，支持按时间倒序排列

### 4.2 增强功能（P1 — 应该有）

- [ ] 批量操作：全选/多选评价，批量通过/拒绝
- [ ] 官方回复：在评价详情内嵌官方回复输入框，提交后显示"官方回复"标记
- [ ] 按条件筛选：按状态/实体类型（圣地/祖庭/行程）/评分筛选
- [ ] Dashboard Badge：首页"待审核评价"卡片显示数量，点击跳转审核列表
- [ ] 日评价量趋势图：折线图展示近 7/30 天每日新增 APPROVED 评价数

### 4.3 优化功能（P2 — 可以有）

- [ ] 搜索评价内容：关键词搜索评价文字
- [ ] 评价详情侧边栏：点击评价展开右侧详情面板（含图片预览），不跳转新页
- [ ] 举报队列：评价举报（ContentReport）与审核流程整合

## 6. API 契约

### 6.1 使用的 API 端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `/api/admin/reviews` | GET | 获取评价列表（分页+筛选） |
| `/api/admin/reviews/:id/status` | PATCH | 审核通过/拒绝/隐藏 |
| `/api/admin/reviews/:id/replies` | POST | 官方回复 |
| `/api/admin/reviews/stats` | GET | 获取统计数据（趋势图+待审核数） |

### 6.2 获取评价列表

#### `GET /api/admin/reviews` （需 Admin JWT）

**请求参数:**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `status` | string | 否 | PENDING / APPROVED / REJECTED / HIDDEN，默认 PENDING |
| `entityType` | string | 否 | HOLY_SITE / TEMPLE / TRIP |
| `rating` | number | 否 | 1-5，筛选特定星级 |
| `page` | number | 否 | 默认 1 |
| `pageSize` | number | 否 | 默认 20，最大 100 |

**响应格式:**

```json
{
  "items": [
    {
      "id": "clxxx",
      "userId": "clyyy",
      "userNickname": "旅行者小明",
      "entityType": "HOLY_SITE",
      "entityId": "clzzz",
      "entityName": "少林寺",
      "rating": 5,
      "title": "震撼心灵的朝圣之旅",
      "content": "这次朝圣...",
      "photos": ["https://..."],
      "status": "PENDING",
      "helpfulCount": 0,
      "createdAt": "2026-03-28T10:00:00Z"
    }
  ],
  "total": 128,
  "page": 1,
  "pageSize": 20
}
```

### 6.3 审核操作

#### `PATCH /api/admin/reviews/:id/status` （需 Admin JWT）

**请求体:**

```json
{
  "status": "REJECTED",
  "reason": "内容不符合平台宗教文化规范"
}
```

**副作用（后端处理）:**
- PENDING → APPROVED: 重新计算 HolySite/Temple 的 avgRating + reviewCount
- PENDING → REJECTED: 发送通知给用户（含拒绝理由）
- APPROVED → HIDDEN: 重新计算 avgRating + reviewCount（减去该评价）

### 6.4 官方回复

#### `POST /api/admin/reviews/:id/replies` （需 Admin JWT）

**请求体:**
```json
{ "content": "感谢您对少林寺的评价！我们将持续提升朝圣者的体验..." }
```

**响应:** 201 Created，返回 ReviewReply 对象（isOfficial: true）

### 6.5 统计数据

#### `GET /api/admin/reviews/stats` （需 Admin JWT）

**请求参数:** `days`（7 或 30，默认 7）

**响应格式:**

```json
{
  "pendingCount": 128,
  "approvedCount": 450,
  "rejectedCount": 23,
  "hiddenCount": 5,
  "dailyTrend": [
    { "date": "2026-03-23", "count": 15 },
    { "date": "2026-03-24", "count": 22 }
  ]
}
```

**错误码:**

| HTTP 状态码 | 错误码 | 说明 |
|------------|-------|------|
| 403 | FORBIDDEN | 非 ADMIN 角色 |
| 404 | NOT_FOUND | 评价不存在 |
| 400 | INVALID_STATUS_TRANSITION | 非法状态转换（如 REJECTED → APPROVED）|

## 7. 数据模型

### 7.1 涉及模型

不新增数据模型，使用 M15 中定义的 Review、ReviewReply 模型。

### 7.2 Admin 页面权限

```typescript
// 所有 Admin 端点需要 @Roles('ADMIN') + JwtAuthGuard + RolesGuard
// 遵循 [R-63] [R-74] 安全铁律
```

### 7.3 状态转换规则

```
允许的审核状态转换:
  PENDING → APPROVED  ✓
  PENDING → REJECTED  ✓
  APPROVED → HIDDEN   ✓
  HIDDEN → APPROVED   ✓ (恢复发布)

禁止的转换:
  REJECTED → APPROVED  ✗ (需用户重新提交)
  APPROVED → REJECTED  ✗ (应使用 HIDDEN)
```

## 8. 技术约束

- **性能要求**: 审核列表首屏加载 < 1.5s；批量审核 20 条 < 3s
- **框架**: Vite + React 19 + Ant Design 5（与现有 Admin 一致）
- **图表**: Recharts（与 Dashboard 现有图表库一致）
- **类型安全**: 禁止 any，ReviewStatus 枚举从 shared-types 导入
- **认证**: 所有 Admin 端点需 JWT + ADMIN 角色（[R-63] [R-74]）
- **并发安全**: 批量审核使用事务处理，防止部分成功导致数据不一致
- **通知触发**: 拒绝评价时自动向用户发送 Notification（type: REVIEW），复用现有 Notification 模型
- **特殊依赖**: 使用已有 Ant Design Table 组件实现列表，Recharts LineChart 实现趋势图
