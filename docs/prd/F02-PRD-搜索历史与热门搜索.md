## 1. 背景与目标

### 1.1 背景

搜索历史和热门搜索是提升搜索效率的两个配套功能。搜索历史让回访用户无需重新输入就能复现上次搜索；热门搜索帮助不确定搜什么的用户发现平台内容热点。这两个功能在 Booking.com、TripAdvisor、Trip.com 等平台上均是标配。当用户打开搜索框但尚未输入时，这是一个关键的"引导时刻"——通过展示历史和热门词，可以显著降低搜索启动阻力。

### 1.2 目标

- [ ] 登录用户的搜索历史在会话间持久化（Redis，最近20条）
- [ ] 未登录用户的搜索历史存 localStorage（最近10条）
- [ ] 热门搜索 TOP10 每小时从 Redis SortedSet 自动刷新
- [ ] 用户可随时清空自己的搜索历史（隐私保护）
- [ ] 搜索框 focus 且无输入时，优先展示历史（上方）+ 热门（下方）

### 1.3 非目标（本期不做的）

- 不包含个性化热门推荐（基于用户信仰偏好，Phase D）
- 不包含热门趋势图表（如"麦加"搜索量趋势，Phase D数据分析）
- 不包含跨设备搜索历史同步（只同步到服务端，不做多端实时推送）
- 不包含历史记录删除单条（只支持清空全部）

## 2. 竞品对标分析

### 2.1 竞品功能对比

| 竞品 | 历史展示方式 | 热门展示方式 | 借鉴点 |
|------|-----------|-----------|-------|
| Booking.com | 搜索框下拉展示"最近搜索"，每条有目的地图标和日期 | "热门目的地"图片卡片横排展示 | 历史每条显示时间戳；热门用图片卡片增强视觉吸引力 |
| TripAdvisor | 焦点时展示"最近查看"（实体卡片，含图片） | "热门目的地"列表，含评分 | 历史展示为实体卡片而非纯文字（体验更好） |
| Trip.com | 搜索框下方"历史搜索"纯文字标签，右上角"×全部清除" | "热门搜索"标签云，按搜索频次加粗 | 标签云设计简洁；全部清除按钮醒目 |
| Agoda | 历史搜索用小图标+地名 | 无明确热门搜索入口 | 历史条目的图标设计 |

### 2.2 差距分析

JOINUS.COM 目前搜索框为空时无任何引导内容，用户面对空搜索框往往不知从何下手。特别是首次访问用户，无任何参照，容易直接关闭搜索。添加热门搜索可以立即改善首次搜索体验。

### 2.3 差异化优势

JOINUS.COM 的热门搜索不仅是通用词，还可以按信仰分类展示热门朝圣地（如"佛教TOP5：少林寺、普陀山、鸡足山..."），这是宗教文化平台独特的热门搜索展示维度。

## 4. 功能清单

### 4.1 核心功能（P0 — 必须有）

- [ ] 搜索历史存储：登录用户存 Redis List，未登录存 localStorage
- [ ] 历史记录展示：搜索框 focus + 无输入时，展示最近 5 条历史（历史标题+时钟图标）
- [ ] 历史条目点击：点击历史词直接执行搜索（等同输入该词后回车）
- [ ] 清空全部历史：历史区域右上角"清空"按钮，确认后清除 Redis/localStorage 数据
- [ ] 热门搜索展示：历史记录下方展示 TOP10 热门词（标签云形式）
- [ ] 热门词点击：点击热门词直接执行搜索
- [ ] 搜索词写入历史：每次成功搜索（跳转搜索结果页）后，将关键词异步写入历史

### 4.2 增强功能（P1 — 应该有）

- [ ] 历史词去重：相同词重复搜索时，只保留最新一条（移动到列表顶部）
- [ ] 历史长度限制：登录用户最多保留20条，localStorage最多10条，超出时删除最旧一条
- [ ] 热门搜索数量标注：热门词后显示搜索次数（如"少林寺 2.3k"）
- [ ] 历史词删除单条：每条历史记录右侧的"×"按钮删除该条（可选，优先级P1）
- [ ] 热门搜索按信仰分Tab：全部 / 佛教TOP5 / 伊斯兰TOP5（移动端展示时）

### 4.3 优化功能（P2 — 可以有）

- [ ] 热门词趋势标记：上升趋势的词显示"🔥"标记
- [ ] 历史词时间戳：每条历史显示搜索时间（"3天前"相对时间）

### 5.4 关键交互流程

```
1. 用户成功搜索（跳转到搜索结果页）
   → 前端异步 POST /api/search/history { keyword: "少林寺" }
   → 登录: Redis LPUSH search:history:{userId} "少林寺"，LTRIM保留20条
   → 未登录: localStorage push，保留10条，去重

2. 用户打开搜索框（无输入）
   → 登录: GET /api/search/history（Redis数据）
   → 未登录: 读取 localStorage
   → 同时 GET /api/search/hot（Redis SortedSet TOP10）
   → 渲染历史（上）+ 热门（下）

3. 用户点击"全部清除"
   → 弹出确认对话框："清空全部搜索历史？"
   → 确认: DELETE /api/search/history（登录）或清空localStorage（未登录）
   → 历史区域消失，保留热门搜索区域

4. 热门搜索更新（后台定时任务，每小时）
   → 扫描 PostgreSQL SearchStat 表，取最高count TOP20
   → 更新 Redis SortedSet search:hot
   → 前端下次 GET /api/search/hot 时获得最新数据
```

### 5.5 三态设计

| 状态 | 触发条件 | 表现 |
|------|---------|------|
| Loading（历史） | 登录用户历史API请求中 | 历史区域显示2行骨架（灰色横条） |
| Empty（无历史） | 用户从未搜索过，或刚清空 | 历史区域不显示（直接展示热门搜索），无"无历史"提示 |
| Error（API失败） | 历史/热门API均失败 | 不显示历史和热门区域，搜索框正常可用（降级为纯输入） |

## 6. API 契约

### 6.1 API 端点总览

| 端点 | 方法 | 用途 | 认证 |
|------|------|------|------|
| `GET /api/search/history` | GET | 获取当前用户搜索历史（最近20条） | JWT必须 |
| `POST /api/search/history` | POST | 记录一条搜索词到历史 | JWT必须 |
| `DELETE /api/search/history` | DELETE | 清空当前用户全部搜索历史 | JWT必须 |
| `GET /api/search/hot` | GET | 获取全站热门搜索TOP10 | 公开 |

### 6.2 搜索历史接口详情

#### `GET /api/search/history`

**响应格式:**

```json
{
  "history": [
    { "keyword": "少林寺", "searchedAt": "2026-03-29T10:23:00Z" },
    { "keyword": "梵蒂冈", "searchedAt": "2026-03-28T15:10:00Z" }
  ]
}
```

> 注：`searchedAt` 从 Redis List 的存储时间戳解析（存储格式：`{keyword}|{timestamp}`）

#### `POST /api/search/history`

**请求体:**

```json
{ "keyword": "少林寺" }
```

**响应格式:**

```json
{ "success": true }
```

**规则:**
- 关键词最大长度 100 字符
- 相同词重复时，删除旧条，LPUSH新条（保持最新在前）
- LTRIM 保留最新20条

#### `DELETE /api/search/history`

**请求体:** 无

**响应格式:**

```json
{ "success": true, "deleted": 5 }
```

### 6.3 热门搜索接口详情

#### `GET /api/search/hot`

**请求参数:**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `limit` | number | 否 | 返回数量，默认10，最大20 |
| `religion` | string | 否 | 按信仰过滤热门（如 buddhism），不传则返回全局热门 |

**响应格式:**

```json
{
  "hot": [
    { "keyword": "麦加", "count": 2340, "trending": true },
    { "keyword": "少林寺", "count": 1890, "trending": false },
    { "keyword": "耶路撒冷", "count": 1560, "trending": false }
  ],
  "updatedAt": "2026-03-29T10:00:00Z"
}
```

**字段说明:**

| 字段 | 说明 |
|------|------|
| `trending` | 近1小时搜索量增长 > 20% 时为 true，前端展示🔥标记 |
| `updatedAt` | 热门榜单最后更新时间，供前端展示"数据截至X时" |

**缓存策略:**
- Redis Key: `search:hot:global`（全局）/ `search:hot:{religion}`（按信仰）
- TTL: 1小时（每小时定时任务刷新）
- 冷启动：若Redis无数据，从 PostgreSQL `SearchStat` 表读取

## 7. 数据模型

### 7.1 Redis 数据结构详细说明

| Key | Redis类型 | 数据格式 | 示例 | TTL |
|-----|---------|---------|------|-----|
| `search:history:{userId}` | List | `{keyword}\|{ISO时间戳}` | `少林寺\|2026-03-29T10:23:00Z` | 30天 |
| `search:hot:global` | SortedSet | member=keyword, score=搜索次数 | - | 1小时（定时刷新） |
| `search:hot:{religionSlug}` | SortedSet | 同上，按信仰过滤 | - | 1小时 |

### 7.2 PostgreSQL SearchStat 表（持久化统计）

```prisma
model SearchStat {
  id        String   @id @default(cuid())
  keyword   String   @unique
  count     Int      @default(0)
  updatedAt DateTime @updatedAt
}
```

**更新策略:**
- 每次搜索：Redis `search:hot` ZINCRBY 1（实时）
- 每小时：定时任务将 Redis SortedSet 前100名写入 SearchStat 表（持久化，防Redis重启数据丢失）

## 8. 技术约束

- **Redis操作**: NestJS SearchModule 直接使用 ioredis 客户端（已有Redis 7 on port 6380）
- **认证**: 历史记录增删改查接口必须有 `JwtAuthGuard`，热门搜索为 `@Public()`
- **隐私合规**: 搜索历史属于个人数据，每个用户只能查询/清除自己的历史（IDOR防护）
- **性能**: 历史接口 P95 ≤ 50ms（纯Redis操作）；热门接口 P95 ≤ 30ms（有缓存时）
- **移动端**: 未登录时使用 `@react-native-async-storage/async-storage` 替代 localStorage
- **小程序**: 微信小程序使用 `wx.setStorage/getStorage` 实现本地历史存储
- **定时任务**: 每小时刷新热门榜单，使用 NestJS `@nestjs/schedule` 的 `@Cron` 装饰器
- **类型安全**: `SearchHistoryItem` 和 `HotKeyword` 接口定义在 `@zuting/shared-types`
