## §3 领域规则

### 五问测试 (BLG扩展版)

```
[BLG-Q1] 目标用户是谁? — 必须能回答
[BLG-Q2] 帮用户做什么决策? — 必须能回答
[BLG-Q3] 去掉这个数据/组件，页面会变差吗? — 否=不属于该页面
[BLG-Q4] 数据来源是真实API还是Mock/硬编码? — Mock=待修
[BLG-Q5] 用户能通过这个页面完成预期任务吗? — 否=功能缺失
```

### 业务逻辑铁律

```
[BLG-01] 页面数据必须服务于目标用户的业务目的
[BLG-02] 三问测试: Q1-Q3任一答不上 = 不属于该页面
[BLG-03] ZUTING业务边界: 见§2表格
[BLG-04] FALLBACK假数据禁止 — 不得用假ID/假URL充数，下游会404
[BLG-05] 空列表≠无数据 — 区分"加载中"/"暂无数据"/"加载失败"
[BLG-06] 价格/日期必须从API获取 — 禁止前端硬编码价格
[BLG-07] 宗教数据准确性 — 信仰/圣地/祖师/祖训的文化内容不得编造
[BLG-08] 状态机合法性 — Trip状态只能按AllowedTransitions跳转 (R-07)
[BLG-09] 货币/单位一致 — 金额统一CNY，距离统一km，时间统一ISO8601
[BLG-10] 分页参数合法 — page≥1, pageSize≤100 (R-64)
[BLG-11] 收藏/评价/订单 必须关联真实用户ID — 禁止匿名写入
[BLG-12] 竞品功能对标 — 每个功能模块至少对标2个竞品 (SCP-02)
```

## §4 执行流程

```
Step 0: 读取目标模块PRD + CLAUDE.md BLG规则
Step 1: 逐页面列出数据项 → 标注数据来源(API/Mock/硬编码)
Step 2: 对每个数据项执行五问测试 (BLG-Q1~Q5)
Step 3: 识别问题:
  - 无业务目的的数据/组件 → 标记移除
  - Mock/硬编码数据 → 标记接API
  - 不符合业务边界 → 标记讨论
Step 4: 修正 (Executor)
  - 移除无目的数据
  - 接入真实API
  - 补充缺失的业务功能
Step 5: 验证 (Verifier)
  - 重跑五问测试
  - 确认API返回真实数据
  - 确认用户可完成预期任务
```

## §5 评分机制

| 维度 | 代号 | 检查项 |
|------|------|--------|
| BLG-D1 数据真实性 | 20% | 所有展示数据来自真实API，无Mock/硬编码 |
| BLG-D2 业务对齐 | 25% | 每个数据项通过五问测试 |
| BLG-D3 功能完整 | 20% | PRD中的用户故事全部可执行 |
| BLG-D4 状态合法 | 15% | 状态机跳转合规，无非法状态 |
| BLG-D5 竞品对标 | 20% | 功能至少达到2个竞品的基准线 |

## §6 工具治理

```
允许工具:
  - Read: 读取页面代码 + PRD + API Controller
  - Grep: 搜索Mock数据/硬编码/TODO/FIXME
  - Glob: 发现页面文件
  - Edit: 修正业务逻辑
  - Bash: 验证API调用 (curl/httpie)

检测命令:
  Mock扫描:     Grep "mock|Mock|MOCK|假数据|placeholder" --glob "*.tsx"
  硬编码价格:   Grep "¥|￥|price.*=.*[0-9]" --glob "*.tsx"
  TODO扫描:     Grep "TODO|FIXME|HACK|XXX" --glob "*.ts" --glob "*.tsx"
  状态机:       Grep "AllowedTransitions|TripStatus" --glob "*.ts"
```
