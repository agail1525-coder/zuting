# 项目++ (Project Generator) v1.0

> **代号**: PJ | **版本**: v1.0 | **创建**: 2026-04-06 | **移植自**: 作业郎 项目++ v2.1
> **定位**: 标准文档自动生成器 — 输入PRD/功能规格 → 输出DOC-XX技术标准 + 施工图 + 验收清单
> **触发**: 新功能开发前 / PRD完成后 / CEO指令

---

## §0 元数据

```yaml
name: project-generator
version: 1.0
triggers:
  - 项目++ {feature-name}    # 指定功能生成标准
  - 新功能 {name}            # 新功能启动流程
前置条件:
  - 对应PRD已存在 (docs/prd/)
  - 竞品调研已完成 (SCP-02)
后置动作:
  - 生成 DOC-XX 标准文档
  - 生成施工图 (管线++)
  - 生成验收清单 (闭环++)
联动技能: 页面++ (消费标准) / 全审++ (验收时引用) / 飞轮++ (闭环检查)
```

## §2 作用域

### 生成物类型

| 类型 | 命名格式 | 存放路径 | 说明 |
|------|---------|---------|------|
| 技术标准 | DOC-{XX}-{名称}-v1.0.md | docs/protocols/ | §0-§12格式 |
| 施工图 | P2-pipeline-{feature}.md | plan/{feature}/ | 变更清单+端点+组件+实施顺序 |
| 验收清单 | P3-checklist-{feature}.md | plan/{feature}/ | N维×条目 验收矩阵 |
| PRD | M/P/F{XX}-PRD-{名称}.md | docs/prd/ | SCP-03格式 |

### 适用场景
- 新模块开发 (如: 二级分销系统)
- 现有模块升级 (如: 消息系统微信级升级)
- 跨端功能对齐 (如: 五端路线预订)
- 技术标准补充 (如: WebSocket通讯标准)

## §3 领域规则

### DOC标准文档格式 (§0-§12)

```
§0  元数据 — 版本/日期/生成方式/适用项目/关联DOC/状态/对标
§1  核心概念与术语
§2  数据模型 — 已有Model + 需新增的Model/字段 + Enum
§3  API契约 — 端点清单 + DTO定义 + 错误码
§4  状态机 — 状态枚举 + 跳转矩阵 + 守卫规则
§5  前端组件 — 组件清单 + Props接口 + 状态管理
§6  WebSocket/实时 — 事件清单 + Payload + 触发场景 (如适用)
§7  安全规则 — Guard + 权限 + 输入校验 + IDOR
§8  i18n — 需新增的翻译key清单 (7语言)
§9  测试要点 — 单元/集成/E2E 关键测试场景
§10 性能预算 — SLA指标 + 缓存策略
§11 部署 — 数据库迁移 + 环境变量 + 依赖
§12 附录 — 竞品截图/参考链接/变更日志
```

### 施工图格式 (管线++)

```
1. Schema变更清单 — 新增字段/Model/Enum表格
2. 新增端点清单 — Method/Path/Feature/DTO表格
3. 新增WebSocket事件 — 事件/Payload/触发场景 (如适用)
4. 新增前端组件 — 组件/类型/位置表格
5. 实施顺序 — 按依赖链排序的Phase 1→N
6. 风险点 — 可能的阻断/兼容性问题
```

### 验收清单格式 (闭环++)

```
N维验收矩阵:
  D1: Schema — Model/字段/Enum是否与DOC一致
  D2: API — 端点是否全部实现 + Swagger文档
  D3: 前端 — 组件是否全部实现 + 页面路由
  D4: 数据流 — API调用→渲染→交互 闭环
  D5: 安全 — Guard/校验/权限 是否符合DOC §7
  D6: i18n — 7语言key是否齐全
  D7: 跨端 — Web/Mobile/Mini/Admin 是否一致
  D8: PRD — 是否满足PRD所有用户故事
```

## §4 执行流程

```
Step 0: 接收需求
  → 确认PRD是否存在 (docs/prd/)
  → 不存在 → 先按SCP-03撰写PRD → 再继续
  → 存在 → 读取PRD + 分析功能范围

Step 1: 竞品调研 (SCP-02)
  → 打开对标竞品页面 (10大旅行网站中相关的)
  → 记录功能点/交互细节/数据结构
  → 输出: 竞品分析摘要

Step 2: 现状扫描
  → Grep/Read 扫描现有代码中已实现的部分
  → 确定: 已有什么 / 缺什么 / 需要改什么
  → 输出: 现状分析 + 差距清单

Step 3: 生成DOC标准文档
  → 按§0-§12格式填充每个Section
  → 关联现有DOC (交叉引用)
  → 标注: 每个§的实现状态 [已有]/[需新增]/[需修改]

Step 4: 生成施工图 (管线++)
  → 从DOC提取变更清单
  → 按依赖链排序Phase
  → 标注每个Phase的预估工作量

Step 5: 生成验收清单 (闭环++)
  → 从DOC §2-§8 提取验收条目
  → 组织为D1-D8维度矩阵
  → 每个条目标注验收方法 (自动/手动)

Step 6: 校验
  → 验证DOC与PRD对齐 (所有用户故事都有对应§)
  → 验证DOC与现有DOC不冲突 (交叉引用正确)
  → 验证施工图的依赖链合理
```

---

## §5 评分机制

项目++的评分针对**生成物质量**:

| 维度 | 权重 | 检查项 |
|------|------|--------|
| PJ-D1 完整性 | 25% | §0-§12每个Section都有实质内容(非占位) |
| PJ-D2 准确性 | 25% | Model/字段名与schema.prisma一致 |
| PJ-D3 可执行 | 20% | 施工图的Phase可直接作为开发任务 |
| PJ-D4 对标深度 | 15% | 竞品功能分析不是泛泛而谈 |
| PJ-D5 交叉一致 | 15% | 与已有DOC/PRD无矛盾 |

## §6 工具治理

```
允许工具:
  - Read: 读取PRD + 现有代码 + 现有DOC
  - Grep: 搜索现有实现 (Model/Controller/Component)
  - Glob: 发现文件结构
  - Write: 生成DOC/施工图/验收清单
  - WebSearch/WebFetch: 竞品调研 (如需要)

禁止:
  - 不在生成阶段修改任何代码
  - 不在生成阶段执行数据库操作
  - 不猜测字段名，必须查schema.prisma确认 (R-59)
```

---

## §7 生命周期

```
L1 触发: CEO指令/新PRD完成/功能升级需求
L2 执行: Step 0-6 调研→分析→生成→校验
L3 输出: DOC-XX + 施工图 + 验收清单 (3件套)
L4 反馈: CEO审阅 → 修改 → 定稿
L5 沉淀: 生成模板优化 / 新的§格式发现 → 更新本技能
```

---

## §8 铁律清单

```
[PJ-01] 无PRD不生成 — PRD不存在时先补PRD (SCP-03)
[PJ-02] 字段名必查 — Model/字段/Enum必须grep schema.prisma确认 (R-59)
[PJ-03] 不猜API — 端点路径必须查Controller确认或明确标注[需新建]
[PJ-04] 交叉引用 — 每个DOC必须列出关联的其他DOC编号
[PJ-05] 实现状态 — 每个§必须标注 [已有]/[需新增]/[需修改]
[PJ-06] 竞品不泛泛 — 必须有具体功能点+交互细节，非"参考Booking"
[PJ-07] 一PRD一DOC — 不允许一个DOC覆盖多个PRD的详细设计
[PJ-08] 版本号必填 — DOC标题必须含v{X}.{Y}版本号
[PJ-09] 生成≠编码 — 项目++只生成文档，不修改代码
[PJ-10] 向后兼容 — 新DOC不能让现有已实现功能变成"需修改"
```
