## 1. 总则：PRD 先行铁律

遵循最高宪法协议 [SCP-03]：

> **无 PRD 不动代码。任何功能模块的开发/重构/升级，必须先完成 PRD 并通过评审，方可开始编码。**

### 1.1 为什么 PRD 先行

| 不写 PRD 的风险 | PRD 先行的收益 |
|----------------|---------------|
| 边做边改，返工严重 | 明确目标，减少无效代码 |
| 竞品优势功能遗漏 | 系统性对标竞品，不遗漏 |
| 前后端接口不匹配 | API 契约提前确认 |
| 验收无据可依 | 可量化的验收标准 |
| 历史决策无记录 | 便于新成员理解背景 |

### 1.2 五步流程（必须按序执行）

```
STEP 1: 深度调研
  └─ 打开对应竞品页面
  └─ 逐一截图、记录功能点、交互细节
  └─ 参考 09-竞品分析规范.md

STEP 2: 撰写 PRD
  └─ 在 docs/prd/ 下创建对应文档
  └─ 按本规范第 3 节模板填写全部章节

STEP 3: PRD 评审
  └─ 自查清单（第 6.2 节）
  └─ CEO++ 审核确认

STEP 4: 方可编码
  └─ PRD 通过后才允许开始代码和 UI/UX 开发
  └─ 编码时随时对照 PRD，发现偏差及时更新

STEP 5: 回溯验证
  └─ 开发完成后，逐项对照 PRD 验收标准检查
  └─ 验收通过后，在 PRD 中标记完成状态
```

## 2. PRD 文件命名与路径

### 2.1 路径结构

```
docs/prd/
├── M{XX}-PRD-{模块名}.md          ← 主模块 PRD（功能集合）
├── P{XX}-PRD-{页面名}.md           ← 子页面 PRD（单个页面）
└── F{XX}-PRD-{功能名}.md           ← 延伸功能 PRD（独立功能）
```

### 2.2 命名规则

| 前缀 | 含义 | 编号范围 | 示例 |
|------|------|---------|------|
| `M` | Module，主模块 PRD | M01 ~ M99 | `M01-PRD-宗教信仰与圣地.md` |
| `P` | Page，子页面 PRD | P01 ~ P99 | `P01-PRD-圣地详情页.md` |
| `F` | Feature，延伸功能 PRD | F01 ~ F99 | `F01-PRD-收藏夹系统.md` |

### 2.3 粒度规则

```
[PRD-G01] 绝不允许一个 PRD 涵盖超过一个页面的详细设计
[PRD-G02] 每个功能模块 → 独立主 PRD（M 系列）
[PRD-G03] 每个子页面 → 独立子页面 PRD（P 系列）
[PRD-G04] 每个独立延伸功能 → 独立功能 PRD（F 系列）
[PRD-G05] 主模块 PRD 列出所有子页面，但每个子页面的详细设计在独立 P 系列 PRD 中
```

## 6. API 契约

### 6.1 使用的 API 端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `/api/holy-sites` | GET | 获取圣地列表 |
| `/api/holy-sites/:id` | GET | 获取圣地详情 |

### 6.2 新增 API 端点（如有）

#### `GET /api/xxx`

**请求参数:**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `page` | number | 否 | 页码，默认 1 |
| `pageSize` | number | 否 | 每页数量，默认 20，最大 100 |

**响应格式:**

```json
{
  "items": [
    {
      "id": "clxxx",
      "name": "圣地名称",
      "nameEn": "Holy Site Name"
    }
  ],
  "total": 60,
  "page": 1,
  "pageSize": 20
}
```

**错误码:**

| HTTP 状态码 | 错误码 | 说明 |
|------------|-------|------|
| 400 | INVALID_PARAMS | 参数校验失败 |
| 404 | NOT_FOUND | 资源不存在 |
| 500 | INTERNAL_ERROR | 服务器内部错误 |

## 4. 各模块内容详解

### 4.1 竞品对标分析（第 2 节）写法

- **必须有截图记录**（存放在 `docs/prd/screenshots/` 目录）
- **不能只写"参考竞品 X"**，必须具体描述借鉴哪个功能点
- **差距分析必须量化**：不写"做得不好"，写"缺少 X 功能，导致用户无法完成 Y 任务"

### 4.2 用户故事（第 3 节）写法标准格式

```
作为一个{用户角色}，
我希望{做某件事}，
以便{获得某种价值}。

验收条件:
- 给定{前提条件}，当{执行操作}，则{期望结果}
- Given {context}, when {action}, then {outcome}
```

### 4.3 API 契约（第 6 节）铁律

- 必须与 Swagger 文档一致（以 Swagger 为准）
- 分页格式必须遵循 [R-05]：`{ items, total, page, pageSize }`
- 错误格式必须遵循 [R-06]：HttpException + HTTP 状态码
- 新增端点必须在此处先定义，再去写代码（合同先行）

### 4.4 验收标准（第 9 节）写法要求

```
[AC-RULE-01] 验收标准必须可测试（有具体操作步骤）
[AC-RULE-02] 验收标准必须有明确边界（通过/不通过的判断标准）
[AC-RULE-03] 边界条件必须覆盖（空数据/超长文本/特殊字符/网络异常）
[AC-RULE-04] 禁止写主观性标准（"界面美观" → "符合 01-UI设计规范.md 中的颜色规范"）
```

## 5. PRD 粒度要求

### 5.1 判断写哪类 PRD 的决策树

```
这是一个全新功能模块（含多个页面）?
  → 是: 先写 M 系列主模块 PRD（概述），再为每个子页面写 P 系列 PRD
  → 否: 这是单个页面的改动?
      → 是: 写 P 系列子页面 PRD
      → 否: 这是一个独立功能点（如收藏、分享、推荐）?
          → 是: 写 F 系列延伸功能 PRD
```

### 5.2 PRD 规模上限

| PRD 类型 | 页面数 | 用户故事数 | 验收标准数 |
|---------|-------|----------|----------|
| M 系列 | 概述，不含页面详情 | ≤ 10 | ≤ 5（整体目标） |
| P 系列 | **恰好 1 个页面** | ≤ 15 | ≤ 20 |
| F 系列 | 跨页面功能 | ≤ 10 | ≤ 15 |

## 6. 评审流程

### 6.1 PRD 评审环节

```
阶段 1: 自查（作者）
  └─ 对照 6.2 节自查清单，全部通过后提交评审

阶段 2: CEO++ 评审
  └─ 核查竞品对标是否充分
  └─ 核查用户故事是否覆盖核心场景
  └─ 核查验收标准是否可量化
  └─ 批准 / 退回修改

阶段 3: 技术可行性确认
  └─ 确认 API 契约无歧义
  └─ 确认数据模型改动合理
  └─ 估算开发工作量

阶段 4: 开发授权
  └─ CEO++ 最终确认
  └─ PRD 状态更新为"已通过"
  └─ 开发人员可以开始编码
```

### 6.2 PRD 自查清单

提交评审前，逐项检查：

- [ ] 文件命名符合 `{编号}-PRD-{名称}.md` 格式
- [ ] PRD 状态已填写
- [ ] 背景和目标清晰，非目标有声明
- [ ] 至少分析了 2 个以上竞品
- [ ] 有截图记录（docs/prd/screenshots/）
- [ ] 用户故事有明确的用户角色
- [ ] 功能清单按 P0/P1/P2 优先级分类
- [ ] 有 ASCII 线框或草图
- [ ] 三态（loading/error/empty）已在线框中体现
- [ ] API 契约符合 [R-04] [R-05] [R-06] 规范
- [ ] 验收标准全部可量化、可测试
- [ ] 无"见竞品截图"这类无效描述

## 7. 验收标准写法

### 7.1 好的验收标准 vs 坏的验收标准

```
❌ 坏: "页面展示圣地列表"
✅ 好: "访问 /holy-sites 页面，应显示至少 20 个圣地卡片，每个卡片包含图片、名称（中英双语）、宗教标签"

❌ 坏: "搜索功能正常工作"
✅ 好: "在搜索框输入'麦加'，点击搜索按钮，1 秒内返回包含'麦加'关键词的圣地列表；若无结果，显示空状态页'暂无相关圣地'和'清空搜索'按钮"

❌ 坏: "移动端显示正常"
✅ 好: "在 Chrome DevTools 模拟 iPhone 14 (390px 宽)，页面无横向滚动条，最小可点击区域不小于 44x44px，文字可读（字号不小于 14px）"

❌ 坏: "加载快"
✅ 好: "Chrome Lighthouse 测试，LCP < 2s，FID < 100ms，CLS < 0.1"
```

### 7.2 边界条件验收标准模板

每个功能必须包含以下边界条件的验收标准：

| 边界场景 | 验收要求 |
|---------|---------|
| 空数据 | 显示 Empty 状态，有引导行动 |
| 超长文本 | 标题/描述超出时有省略号，不破坏布局 |
| 网络错误 | 显示 Error 状态，有重试按钮，点击重试后重新请求 |
| 加载中 | 显示 Skeleton 骨架屏或 Spinner，禁止白屏 |
| 慢网络 | 3G 网速下 10 秒内显示内容或错误 |

## 8. PRD 与代码的双向同步

### 8.1 代码可能比 PRD 更好

PRD 是开发的起点，不是终点。开发过程中如发现 PRD 有改进空间，应**先更新 PRD，再继续开发**。

```
[PRD-SYNC-01] 代码可能比 PRD 更好 → 不盲目删已有好代码，先更新 PRD
[PRD-SYNC-02] 发现 PRD 遗漏 → 立即补充 PRD，不要默默在代码中加功能
[PRD-SYNC-03] 开发中遇到 API 接口需要调整 → 先更新 PRD 中的 API 契约
[PRD-SYNC-04] PRD 与现有代码不一致时 → 先判断哪个更合理，以合理者为准并同步另一方
```

### 8.2 PRD 更新流程

```
发现不一致
  ↓
评估: PRD 更好 还是 代码更好?
  ├─ PRD 更好 → 提 PR 修复代码，保持 PRD 不变
  └─ 代码更好 → 更新 PRD（版本号递增，记录变更），保持代码不变
```

## 9. PRD 更新触发条件

以下情况必须更新对应 PRD：

| 触发条件 | 更新内容 | 紧急程度 |
|---------|---------|---------|
| CEO++ 需求变更 | 目标/功能清单/验收标准 | 立即 |
| 开发中发现 PRD 遗漏 | 补充遗漏的用户故事/验收标准 | 当天 |
| API 接口设计调整 | API 契约章节 | 立即（影响联调）|
| 竞品上线新功能 | 竞品对标分析 | 下次迭代前 |
| 验收发现问题 | 补充边界条件验收标准 | 当次验收后 |
| 数据模型变更 | 数据模型章节 | 立即 |
