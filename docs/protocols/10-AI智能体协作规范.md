## 1. 总则：AI 智能体定位

JOINUS.COM 采用多层次 AI 智能体协作架构，实现人机协同高效开发。AI 智能体是 CEO++ 的执行臂，负责在明确指令和规范约束下完成代码开发、分析、文档等任务。

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **宪法优先** | 所有 Agent 必须无条件遵守 CLAUDE.md 中的全部铁律和协议 |
| **PRD 驱动** | 无 PRD 不动代码，Agent 编码前必须确认对应 PRD 存在且已通过 |
| **人工修改保护** | 人工修改 > AI 判断，任何情况下不得覆盖/还原人工修改 [HE-01] |
| **最小变更** | 只改动完成任务所必需的文件，不做额外"顺手优化" |
| **单任务专注** | 单个 Agent 只做一件事，禁止同时处理 3+ 个独立任务 [C-04] |
| **透明汇报** | 完成波次后必须汇报进度，不得沉默执行超长任务 |

### 1.2 Agent 与 CEO++ 的关系

```
CEO++ (人类)
  ↓ 战略指令 / PRD 审批 / 验收
AI 智能体
  ├─ Heartbeat Daemon  ← 自动周期扫描，发现并执行标准化任务
  ├─ Claude Code       ← 交互式开发，响应即时指令
  └─ 子 Agent (并行)    ← 大任务拆解后的并行执行单元
```

## 2. Agent 类型定义

### 2.1 Heartbeat Daemon（心跳守护进程）

**职责**: 自动周期性扫描项目状态，发现并执行标准化任务，无需人工触发。

```
触发方式: 定时（scripts/heartbeat-daemon.ps1）
扫描范围:
  - TypeScript 编译错误 (tsc --noEmit)
  - 空壳组件检测（HH-V02）
  - 未提交代码警告（B-01）
  - 临时文件清理（B-07）
  - i18n Key 完整性检查

输出:
  - logs/heartbeat/progress.json     ← 当前状态
  - logs/heartbeat/pending-tasks.json ← 待处理任务队列
  - logs/heartbeat/completed-history.json ← 已完成记录
```

**Heartbeat 禁止执行的操作**:
- 修改 .env* 文件
- 执行数据库迁移
- 推送到远端仓库
- 删除任何有实际调用的文件
- 在没有 PRD 的情况下新增功能代码

### 2.2 Claude Code（交互式 AI 开发助手）

**职责**: 响应 CEO++ 的即时指令，执行功能开发、BUG 修复、重构、文档编写等任务。

```
触发方式: CEO++ 直接对话
工作模式:
  - 交互式: 小任务，直接执行并汇报
  - 波次模式 (GOGOGO): 大任务，按波次并行执行并里程碑汇报
  - 规划模式 (plan++): 先出计划，等 CEO++ 确认再执行

上下文:
  - 开始任务前必须读取 CLAUDE.md
  - 使用 Grep 先定位，Read 后精读（C-01/C-02）
  - 不得无脑读全文（大文件带 offset+limit）
```

### 2.3 子 Agent（并行执行单元）

**职责**: 在 GOGOGO 模式下，将大任务拆解后由多个子 Agent 并行执行，互不干扰。

```
并行原则:
  - 每个子 Agent 操作的文件集互不重叠（文件互斥原则 GG-R02）
  - 子 Agent 只能读自己分配的文件，不得越界
  - 完成后汇总到父 Agent 进行合并验证

子 Agent 类型:
  - Scan Agent: 发现任务，生成 pending-tasks.json
  - Build Agent: 执行具体编码任务
  - Verify Agent (YHAT): 验证代码质量，运行类型检查
```

## 3. 任务分配规则

### 3.1 任务分类与分配

| 任务类型 | 负责 Agent | 触发方式 | PRD 要求 |
|---------|-----------|---------|---------|
| 新功能开发 | Claude Code + 子 Agent | CEO++ 指令 | 必须有 PRD |
| BUG 修复 | Claude Code | CEO++ 指令 / Heartbeat 发现 | 不需要 PRD |
| 重构（不改功能） | Claude Code | CEO++ 指令 | 建议有 PRD |
| 文档撰写 | Claude Code | CEO++ 指令 | 不需要 PRD |
| 代码质量扫描 | Heartbeat | 定时触发 | 不适用 |
| 自动清理 | Heartbeat | 定时触发 | 不适用 |
| 竞品分析 | Claude Code | CEO++ 指令 | 不需要 PRD（产出 PRD）|

### 3.2 任务拆解原则

当任务规模较大时（涉及 5+ 文件或 3+ 模块），必须按以下原则拆解：

```
拆解维度 1: 按端拆分
  Wave 1: API 端点实现（services/api/）
  Wave 2: Web 前端实现（apps/web/）
  Wave 3: Mobile 实现（apps/mobile/）
  Wave 4: 小程序实现（apps/miniprogram/）

拆解维度 2: 按功能层拆分
  Layer 1: 数据模型 (Prisma Schema)
  Layer 2: 后端 API (Controller + Service + DTO)
  Layer 3: 前端页面 (UI + API 调用 + 状态管理)

拆解维度 3: 按功能模块拆分
  Module A: 列表页
  Module B: 详情页
  Module C: 创建/编辑表单
```

### 3.3 并行化决策

```
判断是否可以并行:
  Q1: 两个任务会修改同一个文件吗？
      → 是: 必须串行（文件互斥，GG-R02）
      → 否: 可以并行

  Q2: 任务 B 依赖任务 A 的输出吗？
      → 是: 必须串行
      → 否: 可以并行

  Q3: 并行后合并时会有 merge 冲突风险吗？
      → 是: 评估是否值得并行，或调整分工边界
      → 否: 可以并行
```

## 4. Prompt 标准

### 4.1 给 Agent 的任务 Prompt 必须包含

```
[MANDATORY] 每个发给子 Agent 的任务 Prompt 必须包含以下要素:

1. 宪法注入: "遵循 CLAUDE.md 中的全部铁律"
2. 任务边界: "只修改以下文件: [具体文件路径]"
3. 上下文定位: "相关代码位于 [文件路径] 第 [X-Y] 行"
4. 输入规格: "输入数据/接口: [具体描述]"
5. 输出规格: "输出数据/接口: [具体描述]"
6. 质量要求: "完成后必须通过: tsc --noEmit"
7. 禁止操作: "不得修改: [不允许修改的文件]"
```

### 4.2 Prompt 模板

```markdown
## 任务: {任务简短描述}

### 上下文
- 项目: JOINUS.COM — 全球祖庭文化旅行平台
- 规范: 必须遵循 E:\ZUTING\CLAUDE.md 中的全部铁律和协议
- 相关 PRD: docs/prd/{PRD文件名}.md（如有）

### 任务说明
{详细任务描述，1-3 段}

### 文件范围（只能修改以下文件）
- `{文件路径1}` — {修改目的}
- `{文件路径2}` — {修改目的}

### 关键上下文定位
- 相关接口定义: `{文件路径}` 第 {X} 行，`{函数名/接口名}`
- 相关组件: `{文件路径}` 第 {Y} 行
- Prisma Schema 中的相关 Model: `services/api/prisma/schema.prisma` 第 {Z} 行

### 输出要求
- {具体输出要求1}
- {具体输出要求2}
- 禁止使用 any 类型 [R-01]
- 禁止硬编码文本（使用 i18n Key）[I18N-F01]

### 质量门禁
完成后必须确认:
- [ ] TypeScript 编译零错误 (`tsc --noEmit`)
- [ ] 三态（loading/error/empty）全部处理
- [ ] 无 console.log 残留（调试代码清理）

### 禁止修改
- 不得修改: {列出不允许碰的文件}
- 不得删除: {列出不允许删除的代码/功能}
```

### 4.3 精准上下文传递原则

遵循 [C-05]：拆任务时提供精确上下文（文件+行号），避免 Agent 重新探索。

```
❌ 低效 Prompt: "修复 mobile 的首页显示问题"
   → Agent 需要自己找文件，浪费大量上下文

✅ 高效 Prompt: "修复 apps/mobile/app/(tabs)/index.tsx 第 87 行的字段映射：
                API 返回 name 字段，但代码使用了 nameZh，导致显示空白"
   → Agent 直接定位，高效执行
```

## 5. GOGOGO 七铁律

GOGOGO 是大规模并行任务执行模式的触发指令，执行时必须严格遵守以下七铁律：

### [GG-R01] TypeScript 零错误门禁

```
每个波次完成后，必须运行 tsc --noEmit 确认零编译错误。
有错误不得提交，不得进入下一波次。
```

### [GG-R02] 文件互斥原则

```
并行执行的子任务，操作的文件集必须完全互斥。
同一文件在同一时刻只能有一个 Agent 修改。
违反此规则会导致 merge 冲突，直接拒绝执行。
```

### [GG-R03] 最小变更原则

```
只修改完成任务所必需的代码。
禁止"顺手优化"无关代码（即使看起来更好）。
不确定是否需要改动 → 不改，先询问 CEO++。
```

### [GG-R04] PRD 先行检查

```
每个波次开始前，确认所有任务都有对应的已通过 PRD。
如 PRD 不存在 → 停止执行，先请 CEO++ 创建 PRD。
如 PRD 未通过评审 → 停止执行，等待 PRD 评审通过。
```

### [GG-R05] 波次里程碑汇报

```
每个波次完成后，必须输出汇报：
  - 本波次完成了什么
  - 哪些文件被修改
  - tsc 检查结果
  - 是否有需要 CEO++ 决策的问题
  - 下一波次计划

禁止沉默执行超过 3 个波次不汇报。
```

### [GG-R06] 人工修改保护

```
执行前检查 git diff，识别人工修改的痕迹。
发现人工修改的代码 → 绝对不覆盖/还原。
遇到冲突 → 停止，向 CEO++ 汇报，等待指令。
```

### [GG-R07] 提交原子性

```
每个完整功能单元完成后立即 git add + git commit（B-01）。
禁止积累多个波次的改动后一次性提交。
单次 commit ≤ 30 个文件（B-02）。
```

## 6. 质量门禁系统

### 6.1 五层质量验证（V1-V5）

对应 CLAUDE.md 中的全审++ (hohoho) 体系：

| 层级 | 名称 | 检查内容 | 工具 |
|------|------|---------|------|
| **V1** | 代码质量 | tsc 编译/ESLint/命名规范/死代码 | `tsc --noEmit`, ESLint |
| **V2** | 功能完整 | API 连通/CRUD 完整/状态机/数据验证 | 手动测试 / API 测试 |
| **V3** | 用户体验 | loading/error/empty 三态/响应式/i18n | 浏览器测试 |
| **V4** | 安全 | 认证 Guard/输入验证/IDOR 防护/CORS | 代码审查 |
| **V5** | 性能 | bundle 大小/lazy loading/图片/缓存 | Lighthouse |

### 6.2 各阶段门禁要求

| 阶段 | 最低通过标准 |
|------|------------|
| 开发中 | V1 通过（编译零错误）|
| 提交 PR/Commit | V1 + V2 通过 |
| 上线发布 | V1 + V2 + V3 + V4 通过 |
| 重大版本 | V1 ~ V5 全部通过 |

### 6.3 V1 代码质量自动检查脚本

```bash
# 在项目根目录运行全量类型检查
pnpm --filter @zuting/api tsc --noEmit
pnpm --filter @zuting/web tsc --noEmit
pnpm --filter @zuting/admin tsc --noEmit
pnpm --filter @zuting/mobile tsc --noEmit
pnpm --filter @zuting/miniprogram tsc --noEmit

# 快捷命令（如在 package.json 中配置）
pnpm typecheck:all
```

### 6.4 YHAT 验证 Agent

YHAT（Yet Another Heartbeat Agent Testing）是专门执行质量验证的子 Agent：

```
YHAT 触发时机:
  - 每个波次完成后
  - 合并 PR 前
  - 部署前

YHAT 执行清单:
  1. 运行 tsc --noEmit（所有端）
  2. 检查新增文件是否有对应调用者（HH-V04）
  3. 检查是否存在 any 类型新增（R-01）
  4. 检查 console.log 是否清理
  5. 检查三态是否全部实现
  6. 输出 YHAT 报告
```

## 7. 协作流程 SOP

### 7.1 标准开发任务流程

```
CEO++ 发出指令
  ↓
[Claude Code] 读取 CLAUDE.md（必须步骤）
  ↓
[Claude Code] 确认是否有对应 PRD
  ├─ 无 PRD → 停止，提示 CEO++ 先创建 PRD
  └─ 有 PRD → 继续
  ↓
[Claude Code] Grep 定位相关代码，Read 精读
  ↓
[Claude Code] 制定执行计划（plan++）
  ↓
CEO++ 确认计划（或直接 GOGOGO 跳过规划）
  ↓
[Claude Code] 按波次执行（子 Agent 并行）
  ↓
每波次完成: 里程碑汇报 + tsc 检查
  ↓
[YHAT] 全量质量验证
  ↓
git commit（遵循 B-01~B-03）
  ↓
CEO++ 验收
```

### 7.2 BUG 修复流程

```
发现 BUG（CEO++ 报告 / Heartbeat 发现）
  ↓
[Claude Code] 定位根因
  1. 读取错误日志/报错信息
  2. Grep 搜索相关代码
  3. Read 精读问题代码段（带 offset+limit）
  ↓
[Claude Code] 分析根因，制定修复方案（简短描述）
  ↓
CEO++ 确认修复方案
  ↓
最小化修复（只改必要代码，GG-R03）
  ↓
tsc --noEmit 确认编译通过
  ↓
git commit -m "fix({scope}): {描述}"
```

### 7.3 CEO++ 指令速查（Agent 识别表）

| 指令 | Agent 行为 |
|------|----------|
| `GOGOGO` | 进入全速并行模式，按七铁律执行，里程碑汇报 |
| `KKKK` | 断点恢复，从最后已完成的波次继续执行 |
| `plan++` | 先输出详细执行计划，等 CEO++ 确认后再执行 |
| `页面++` | 对指定页面执行 PRD 驱动深度重构 |
| `hohoho` | 对指定模块执行全审（V1-V5）并修复发现的问题 |
| `X+` | 自动 BUG 猎杀，全项目扫描并修复已知类型的 BUG |
| `Y+` | 宪法审查，检查代码是否符合 CLAUDE.md 全部铁律 |
| `保存++` | 五层沉淀：memory + CLAUDE.md + protocols + git + docs |
| `PUSH` | git add + commit + push |
| `P+` | 保存++ + PUSH++（保存并部署）|

## 8. 上下文管理规范

遵循 CLAUDE.md [C-01] ~ [C-05] 铁律：

### 8.1 [C-01] 禁止无脑读全文

```
Read 工具必须带 offset + limit，除非文件 < 100 行。

❌ Read(file_path="services/api/prisma/schema.prisma")
✅ Read(file_path="services/api/prisma/schema.prisma", offset=1, limit=50)
   + 先 Grep 定位目标 Model，再精读对应行范围
```

### 8.2 [C-02] 先 Grep 后 Read

```
查找代码时的标准流程:

Step 1: Grep 搜索关键字
  Grep(pattern="HolySite", path="services/api/src")
  → 获得文件路径和行号

Step 2: Read 精读目标区域
  Read(file_path="xxx.ts", offset=45, limit=30)
  → 只读必要内容

禁止: 直接 Read 整个文件找代码
```

### 8.3 [C-04] 单 Agent 单任务

```
每个 Agent 实例只负责一件事。
禁止同时处理 3+ 个独立功能模块。

如果任务确实很大 → 拆分成多个子 Agent 并行执行，每个子 Agent 只负责一个模块。
```

### 8.4 [C-05] 精确上下文传递

```
向子 Agent 传递任务时，必须提供:
  - 精确的文件路径（绝对路径）
  - 相关代码的行号范围
  - 直接相关的接口/类型定义位置

禁止: "修改相关的代码"（模糊，浪费上下文）
要求: "修改 E:\ZUTING\apps\mobile\app\(tabs)\index.tsx 第 82-96 行的 renderHeroSection 函数"
```

### 8.5 上下文窗口保护

```
长任务执行时，保护上下文窗口的规则:

[CTX-01] 每 5000 Token 输出一次简要进度汇报（防止遗忘前期状态）
[CTX-02] 读取大文件时，分段读取（每次最多 200 行）
[CTX-03] Grep 结果超过 50 条时，缩小搜索范围后重新搜索
[CTX-04] 波次间可通过 memory/ 系统持久化关键状态（跨会话恢复）
```

## 9. 记忆系统规范

### 9.1 记忆文件结构

```
memory/
├── MEMORY.md                      ← 主索引（所有记忆文件的目录）
├── user_ceo_plus.md               ← CEO++ 角色定义
├── feedback_commands.md           ← ++指令体系完整定义
├── feedback_autonomous.md         ← 自主执行风格偏好
├── reference_server_deploy.md     ← 服务器部署信息
├── feedback_nginx_docker.md       ← Nginx/Docker 配置铁律
├── project_*.md                   ← 项目里程碑记录
├── feedback_*.md                  ← 从历史教训提取的规范
└── reference_*.md                 ← 长期参考信息
```

### 9.2 记忆文件命名规范

| 前缀 | 用途 | 内容类型 |
|------|------|---------|
| `user_` | 用户偏好/角色定义 | CEO++ 指令系统、执行风格 |
| `project_` | 项目里程碑/状态 | 功能完成记录、重大决策 |
| `feedback_` | 从教训中提取的规范 | 踩坑记录、防错规则 |
| `reference_` | 长期参考信息 | 服务器信息、配置值 |

### 9.3 记忆文件维护规则

```
[MEM-01] 每次会话中发现新的重要规范/踩坑 → 写入对应 memory/*.md
[MEM-02] MEMORY.md 必须实时维护索引，新增文件立即更新索引
[MEM-03] 记忆文件只增不删（可标注"已废弃"，但不删除历史决策）
[MEM-04] 每次任务开始前读取 MEMORY.md，确保了解最新上下文
[MEM-05] 跨会话持久化: 重要状态写入 memory/，不依赖会话内记忆
```

### 9.4 保存++ 执行流程

CEO++ 触发 `保存++` 时，执行五层沉淀：

## 11. 常见问题排查

### 11.1 Agent 执行失败常见原因

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| tsc 编译报错 | any 类型/接口不匹配/枚举不一致 | 检查 [R-01][R-02]，逐错修复 |
| 找不到相关文件 | 路径错误/文件被移动 | 先 Glob 搜索确认路径 |
| API 返回 404 | 路由未注册/Controller 路径错误 | 检查 NestJS 模块导入 |
| Prisma 字段报错 | 字段名猜测错误（R-59 铁律） | Read schema.prisma 确认字段名 |
| 前端显示空白 | API 字段名不匹配 | 对比 API 响应与前端取值 |
| 组件未显示 | 新文件没有调用者（HH-V04）| 确认组件被正确引入 |

### 11.2 KKKK 断点恢复 SOP

当任务中断，CEO++ 触发 `KKKK` 时：

```
Step 1: 读取 logs/heartbeat/progress.json 或 memory/（确认上次完成到哪里）
Step 2: 运行 git status + git log --oneline -5（确认当前代码状态）
Step 3: 运行 tsc --noEmit（确认当前编译状态）
Step 4: 对比已完成波次 vs 总计划，确定断点
Step 5: 从断点继续执行，跳过已完成的部分
Step 6: 输出恢复汇报：已完成/剩余/计划
```

### 11.3 任务被卡住的处理

```
以下情况 Agent 必须停止并向 CEO++ 汇报，不得自行决策:

1. 发现需要修改超出指定范围的文件
2. 发现 PRD 与现有代码存在根本冲突
3. 发现安全漏洞或数据完整性风险
4. 遇到三个以上的 TypeScript 错误且不清楚根因
5. 需要安装新的 npm 依赖
6. 需要修改 Prisma Schema（涉及数据库变更）
7. 遇到人工修改保护（HE-01）的代码需要修改
```
