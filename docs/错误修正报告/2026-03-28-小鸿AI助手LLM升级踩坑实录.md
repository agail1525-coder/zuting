## 1. 事故概要

小鸿AI助手原为纯规则引擎（关键词匹配+数据库查询+模板拼接），不调用任何大模型。本次升级目标：接入小白GPU服务器上的 Qwen3.5-35B 大模型（vLLM），实现真正的LLM智能回答。

升级过程中踩了7个坑，其中3个是阻断级问题。

## 2. 踩坑清单

### [E-XH-01] 设计方向错误：最初按Claude API设计，实际需要用Qwen vLLM
- **严重程度**: P0 阻断
- **表现**: 安装了 `@anthropic-ai/sdk`，代码用Anthropic SDK写法，完全不匹配实际部署架构
- **根因**: AI助手默认思维是用Claude API，没有读取小白服务器部署文档确认实际架构
- **修复**: 完全重写 `xiaohong.service.ts`，移除Anthropic SDK，改用原生 `fetch()` 调用OpenAI兼容格式API
- **教训**: **先确认目标架构再写代码**。CEO明确说了"直接用小白服务器上的QWEN3.5大模型"

### [E-XH-02] 模型路径错误：9B vs 35B vs 实际部署
- **严重程度**: P1
- **表现**: 部署文档记录模型为 `qwen3.5-9b`，CEO说是35B，实际vLLM返回 `qwen3.5-35b-a3b-fp8`
- **根因**:
  1. 部署文档(2026-03-27)记录的是初始部署时的9B模型
  2. 之后小白服务器已升级到35B模型但文档未更新
  3. 模型全名是 `Qwen3.5-35B-A3B-FP8`（MoE稀疏模型，35B总参数，3B激活参数，FP8量化）
- **修复**: `curl -H "Authorization: Bearer xxx" http://120.24.31.151:18001/v1/models` 直接查询vLLM获取真实模型ID
- **教训**: **不信文档，信运行时** — 部署文档是时间点快照，永远通过API确认实际状态
- **影响文件**: `.env`, `.env.example`, `seed.ts`, `AIConfigPage.tsx`

### [E-XH-03] Prisma Generate EPERM：Windows DLL锁定
- **严重程度**: P1 阻断
- **表现**: `prisma generate` 报错 `EPERM: operation not permitted, rename query_engine-windows.dll.node`
- **根因**: API服务器正在运行，持有 `query_engine-windows.dll.node` 文件句柄，Windows不允许重命名被占用的文件
- **修复**: 先 `taskkill` 所有NestJS相关进程，再执行 `prisma generate`
- **教训**: **Windows上修改Prisma schema后，必须先停API再generate**。Linux不存在此问题
- **防御**: 可考虑在 `package.json` 中添加 `pregenerate` 脚本自动停服务

### [E-XH-04] Qwen3.5 Thinking Mode：输出包含推理过程
- **严重程度**: P2
- **表现**: 模型回复以 `Thinking Process:\n1. **Analyze the Input:**...` 开头，用户看到大段推理垃圾
- **根因**: Qwen3.5默认启用thinking/reasoning模式（类似DeepSeek R1），会先输出思维链
- **修复**:
  1. 请求体添加 `chat_template_kwargs: { enable_thinking: false }` 禁用thinking
  2. 后备：`stripThinkingPrefix()` 方法清除 `<think>...</think>` 标签和 `Thinking Process:` 前缀
- **教训**: **新模型必须先直接curl测试**，确认输出格式再写代码
- **代码位置**: `xiaohong.service.ts:callLLM()`, `stripThinkingPrefix()`

### [E-XH-05] LLM调用超时：60s对35B模型+RAG上下文不够
- **严重程度**: P1 阻断
- **表现**: 每次chat调用都60s超时，然后降级到规则引擎回答
- **根因**:
  1. 原始超时设60s，但35B模型+完整RAG上下文（含系统提示词+安全约束+数据库检索结果）生成时间约15-40s
  2. 加上网络延迟（小白服务器在AutoDL，经FRP隧道中转），首次调用可能更慢
  3. 如果vLLM刚启动需要warm up，首次推理更慢
- **修复**: 非流式超时调至180s，流式调至300s
- **教训**: **大模型超时要预留充分余量**，特别是经过隧道的远程调用

### [E-XH-06] Windows终端编码：中文消息在curl中乱码
- **严重程度**: P2
- **表现**: 服务器日志显示 `Intent: GENERAL for: "���С��"`，中文消息完全乱码
- **根因**: Git Bash on Windows默认使用GBK/CP936编码，直接在 `-d '{"message":"中文"}` 中写中文会被编码转换破坏
- **修复**: 使用 `printf '{"message":"中文"}' > /tmp/test.json && curl -d @/tmp/test.json` 方式传递UTF-8内容
- **教训**: **Windows终端测试中文API时，用文件传参而非内联字符串**
- **注意**: 这只影响开发测试，不影响前端调用（前端用JSON.stringify正确处理UTF-8）

### [E-XH-07] Admin页面残留Claude选项
- **严重程度**: P3
- **表现**: AI配置中心模型选择器显示 `Claude Sonnet 4`, `Claude Opus 4`, `Claude Haiku 4`
- **根因**: 从ZuoYeLang项目移植AI配置页面时，未更新模型选项为Qwen
- **修复**: Select options改为 `Qwen3.5-35B-A3B-FP8` 和 `Qwen3.5-122B-A10B-FP8`
- **教训**: **移植代码时必须全文搜索旧品牌名**，一处残留就是一处迷惑

## 3. 架构决策记录

### 为什么用原生fetch而非OpenAI SDK？
- vLLM的OpenAI兼容API并非100%兼容（如 `chat_template_kwargs` 是vLLM扩展字段）
- 原生fetch零依赖，Node.js 18+内置
- 减少一个npm包 = 少一个升级风险点
- 代码量差异极小（~30行 vs SDK也差不多）

### 为什么chat端点是@Public()？
- 未登录用户也需要体验AI（引流转化）
- 登录用户通过 `req.user?.id` 可选保存对话历史
- 未登录用户使用降级模式（无历史记录、无conversationId）

### RAG策略
- 意图检测 → 关键词匹配（快、不耗token）
- 数据检索 → Prisma查询（精确、可控）
- 上下文注入 → 拼接到system prompt末尾
- 为什么不用向量数据库？数据量196条，结构化数据用SQL检索更精确

## 8. 总结

本次升级核心难点不在代码量（~600行新增），而在**架构对齐**：
1. 必须匹配实际GPU服务器的模型和API格式
2. 必须处理Qwen3.5特有的thinking mode
3. 必须适配Windows开发环境的各种限制
4. 必须保证降级路径（LLM不可用时自动切规则引擎）

最终效果：小鸿从一个"关键词匹配机器人"变成了真正能理解问题、引用平台数据、给出专业回答的AI助手。
