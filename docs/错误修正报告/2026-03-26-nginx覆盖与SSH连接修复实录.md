# 2026-03-26 Nginx配置覆盖与SSH连接修复实录

> 严重程度: **P0** | 影响范围: 作业郎生产环境全站不可用 + SSH管理通道中断
> 修复耗时: ~2小时 | 修复人: Claude + CEO

---

## 事故概要

ZUTING项目首次部署时，nginx-standalone.conf被改写，导致:
1. 直连IP(120.24.31.151)不再显示作业郎后台登录页
2. api.fszyl.top路由丢失
3. 作业郎管理后台被挤到 `/admin` 子路径

同时修复SSH时触发了fail2ban封禁 + Clash TUN流量劫持双重障碍。

---

## 故障一: Nginx配置覆盖

### 根因
ZUTING部署将 `location /` 默认路由指向了祖庭Web(172.19.0.1:3003)，覆盖了原本指向作业郎webadmin的配置。

### 影响文件
- `/opt/zuoyelang/docker/nginx/nginx-standalone.conf` (Docker nginx挂载的唯一配置)

### 错误配置 (被覆盖后)
```nginx
# 默认server块
location / {
    proxy_pass http://172.19.0.1:3003;  # ← 错！指向了ZUTING
}
```

### 修复方案
1. 恢复默认server的 `location /` 指向作业郎webadmin
2. 新增独立server块处理zuting.fszyl.top域名

```nginx
# 默认server (直连IP + api.fszyl.top)
server {
    listen 80 default_server;
    location / {
        proxy_pass http://webadmin;  # ← 恢复作业郎
    }
    # ... api.fszyl.top的API路由保持不变
}

# 新增: ZUTING独立server块
server {
    listen 80;
    server_name zuting.fszyl.top;
    location / {
        proxy_pass http://172.19.0.1:3003;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 踩坑点
- Docker nginx容器内部访问宿主机服务需要用Docker gateway IP `172.19.0.1`，不是 `localhost`
- ZUTING的Web(3003)和API(3002)都跑在宿主机上，不在Docker内

## 故障二: SSH连接中断 (Clash TUN劫持)

### 根因
本地Clash Meta(mihomo-party)开启了TUN模式(`auto-route: true`)，所有TCP流量被劫持到虚拟网卡`Mihomo`，SSH连接被代理服务器转发而非直连。

### 症状链
1. 首次SSH尝试 → 通过代理 → 认证方式不匹配 → "Permission denied"
2. fail2ban检测到失败 → 封禁出口IP `168.143.110.4`
3. 后续所有SSH → "Error reading SSH protocol banner" (连TCP握手都被拒)

### 诊断过程
```python
# 关键诊断: 检查本地socket绑定地址
sock = socket.create_connection(('120.24.31.151', 22))
print(sock.getsockname())  # → ('198.18.0.1', xxxxx)
# 198.18.0.1 是mihomo fake-ip范围! 证实流量走了TUN
```

### 修复步骤
1. **解封fail2ban** (需用阿里云VNC控制台，因为SSH不通):
   ```bash
   fail2ban-client set sshd unbanip 168.143.110.4
   ```

2. **mihomo.yaml 添加路由排除** (永久生效):
   ```yaml
   tun:
     route-exclude-address:
       - 120.24.31.151/32
   ```
   文件位置: `C:\Users\HONG\AppData\Roaming\mihomo-party\mihomo.yaml`

3. **work/config.yaml 添加直连规则** (运行时配置):
   ```yaml
   rules:
     - IP-CIDR,120.24.31.151/32,DIRECT
   tun:
     route-exclude-address:
       - 120.24.31.151/32
   ```

4. **重启mihomo内核**: 右键托盘图标 → "重启内核"

### 踩坑点
- mihomo有两层配置: `mihomo.yaml`(用户层) + `work/config.yaml`(运行时)，两处都要改
- 改完config后必须重启内核，热更新不生效
- 第一次重启内核后config被覆写回旧版，需要再次编辑+再次重启
- fail2ban一旦封禁，只能通过VNC控制台或其他未被封禁的IP解封
- `sshpass`在Windows上不可靠，必须用Python `paramiko`做SSH

---

## 故障三: 部署脚本缺失导致手动操作出错

### 根因
ZUTING项目缺少自动化部署脚本，手动操作nginx配置时覆盖了作业郎的设置。

### 修复方案
创建 `E:\ZUTING\scripts\deploy.py`，统一管理两个项目的部署:
- 自动检测当前项目(ZUTING/ZuoYeLang)
- paramiko SFTP上传 + 远程重启
- 不触碰nginx配置(nginx由独立server块隔离)

---

## 铁律总结

| # | 铁律 | 原因 |
|---|------|------|
| 1 | **永远不要修改nginx默认server块的 `location /`** | 它是作业郎的入口，改了全站挂 |
| 2 | **新项目用独立 `server_name` 块** | 域名隔离，互不影响 |
| 3 | **SSH必须走paramiko，不用sshpass** | Windows + Clash TUN环境下sshpass不可靠 |
| 4 | **mihomo.yaml的route-exclude-address必须包含服务器IP** | 否则SSH被TUN劫持 |
| 5 | **fail2ban封禁后只能VNC解封** | SSH通道已断，无法远程操作 |
| 6 | **部署用脚本不用手动** | 手动操作是配置覆盖的根源 |
